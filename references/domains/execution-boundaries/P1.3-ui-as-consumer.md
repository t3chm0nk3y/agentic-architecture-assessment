## Intent

The UI layer must consume the API only. It must not call runtime, orchestration, or integration components directly. All interaction between the UI and the agent system must pass through the defined API boundary.

## Why It Matters

When the UI layer imports or calls backend runtime modules directly, the API becomes an incomplete contract. The system now has two communication paths: the API and the direct coupling. Changes to the runtime break the UI in ways the API contract cannot predict or protect against. The API cannot be versioned, rate-limited, or secured independently because some operations bypass it. Other consumers (mobile clients, third-party integrations, CLI tools) cannot replicate the UI's behavior because they only have access to the API, not the direct imports. In production, this creates a class of bugs visible only in the UI, inconsistent behavior across clients, and an API that does not represent the system's actual capabilities.

## Applicability

Applies to: All system types with a UI layer (web frontend, desktop application, mobile client)
Requires: A user interface that interacts with agent execution or results

## N/A Conditions

This principle is N/A when the system has no UI layer. Examples: a headless API service, a CLI-only tool, a library or SDK consumed programmatically, or a system that is operated exclusively through API calls or configuration files.

## Required Behaviors

1. The UI layer communicates with the backend exclusively through a defined API (HTTP endpoints, GraphQL, WebSocket protocol, or equivalent network boundary)
2. No UI module imports or directly references backend runtime, orchestration, or integration modules
3. The UI uses an API client layer (HTTP client, SDK, generated client) to interact with the system, not direct function calls
4. Data flowing from the backend to the UI passes through the API response layer — the UI does not access internal data structures, database connections, or in-memory state directly
5. Any real-time data (streaming events, status updates) reaches the UI through a defined streaming protocol (SSE, WebSocket, long-polling), not through direct event bus subscriptions

## Acceptable Implementation Patterns

- A single-page application that communicates with the backend exclusively through REST or GraphQL endpoints
- A server-rendered application where the rendering layer consumes the same API that external clients use
- A micro-frontend architecture where each frontend module communicates with its corresponding backend service through the service's public API
- A desktop or mobile application that uses an HTTP client or SDK to interact with the agent system
- A WebSocket-based UI that subscribes to events through a defined WebSocket protocol, with the WebSocket server acting as the API boundary
- A generated API client (from OpenAPI, GraphQL schema, or similar) used by the UI, ensuring the UI interacts only with the documented API surface

## Evidence to Seek

- **Runtime evidence:** Network traffic between UI and backend showing all interactions go through API endpoints, with no direct module loading or in-process calls
- **Trace evidence:** Follow the UI's data fetching code from component to API call — verify each path reaches an HTTP/WebSocket endpoint, not a backend module import
- **Implementation evidence:** UI module imports showing only API client libraries, fetch utilities, or SDK imports — no imports from backend runtime, orchestration, or integration packages
- **Declarative evidence:** Architecture documentation describing the UI as an API consumer (lowest weight)

## Trace Method

1. Identify the UI layer (frontend application directory, client-side code, rendering modules)
2. Examine the UI's dependency declarations (package manifests, import maps) for any references to backend packages or modules
3. Find the UI's data fetching layer (API client, service layer, data hooks, store/state management that calls the backend)
4. Sample 3-5 UI data fetching call sites — trace each from the UI component through to the actual call target
5. Verify that call targets are API endpoints (URLs, endpoint paths) rather than direct module imports
6. Search for direct imports of backend modules in UI code: look for imports of runtime, orchestration, engine, or tool modules from backend package paths
7. Check for shared code between UI and backend — shared types or interfaces are acceptable; shared runtime logic is a boundary violation

## Verdict Conditions

**PASS when:** The UI layer communicates exclusively through a defined API boundary AND no UI module imports backend runtime, orchestration, or integration modules AND all sampled data fetching paths terminate at API endpoints or client SDK calls.

**PARTIAL when:** The UI primarily uses the API but has isolated direct references to backend modules (e.g., one shared utility that exposes runtime internals to the UI), OR the UI uses the API for most operations but subscribes to internal event buses for real-time updates instead of using a streaming protocol through the API.

**GAP when:** The UI routinely imports backend runtime or orchestration modules, OR the UI bypasses the API for core operations (run creation, step management, result retrieval), OR the UI directly accesses backend data stores or in-memory state.

**UNASSESSABLE when:** The UI framework abstracts the communication layer to a degree that prevents determining whether API endpoints or direct imports are used (e.g., a full-stack framework with magic imports that may or may not cross a network boundary), and no runtime inspection is available.

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether a build system bundles backend code into the UI in ways not visible from source imports (e.g., aliased imports that resolve to backend modules at build time). However, inspecting import statements and dependency declarations in source code is sufficient for assessment in standard setups.

## Related Principles

- [P1.1](P1.1-execution-centralization.md) — If the UI calls the runtime directly, it becomes an execution entry point outside the centralized runtime, violating both P1.3 and P1.1
- [P1.2](P1.2-transport-runtime-separation.md) — P1.3 depends on the API being a clean boundary. If the transport layer owns execution logic (P1.2 violation), the UI consuming that API inherits execution coupling indirectly.
- [P1.4](P1.4-execution-errors-separate.md) — The UI's ability to correctly handle execution failures depends on those failures being surfaced through the API as run state, not as HTTP errors. P1.3 ensures the UI receives errors through the right channel; P1.4 ensures those errors are correctly categorized.

## Boundary Notes

- Shared type definitions (interfaces, enums, data transfer objects) between UI and backend do not violate P1.3. The concern is shared runtime logic, not shared type contracts. A monorepo that shares a `types` package between frontend and backend is acceptable as long as the types package contains no execution logic.
- Server-side rendering (SSR) introduces a gray area: the rendering server may import backend modules to pre-fetch data. This is acceptable if the SSR layer acts as an API consumer (calling API endpoints internally) rather than importing runtime modules directly. If the SSR layer calls the runtime directly, it is functioning as a second backend, not a UI consumer.
- In full-stack frameworks that blur the client/server boundary (e.g., frameworks with server functions callable from client components), assess whether the framework maintains a network boundary between client and server code. If the framework serializes calls across a network boundary, the principle is satisfied. If client code can call server functions in-process, the principle may be violated.
- Build-time code sharing (e.g., isomorphic modules) is acceptable for utilities like date formatting or validation. It is a violation if the shared code includes execution, orchestration, or integration logic.
