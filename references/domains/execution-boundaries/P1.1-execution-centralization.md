## Intent

All agent execution logic must live in a single, identifiable runtime module. No other module may initiate runs, drive step sequencing, or own the agent loop. Every entry point in the system — API handlers, CLI commands, scheduled tasks, message consumers — must delegate to this single runtime for execution.

## Why It Matters

When execution logic is spread across multiple modules, the system has multiple sources of truth for how an agent run behaves. Each copy of the execution loop may handle edge cases differently, apply different retry policies, emit different events, or fail in different ways. Operators cannot predict system behavior because it depends on which entry point triggered the run. Modifications to execution logic require changes in multiple places, and missed locations silently diverge. In production, this manifests as inconsistent run outcomes, missing observability for some execution paths, and bugs that appear only through specific entry points.

## Applicability

Applies to: All system types with more than one module
Requires: A codebase with module separation (multiple files, packages, or services)

## N/A Conditions

This principle is N/A when the system is a single-file script with no module separation. If the entire system lives in one file, there is no architectural boundary to assess — execution is centralized by definition.

## Required Behaviors

1. A single, identifiable module owns the agent execution loop (run initiation, step sequencing, completion detection)
2. All entry points (API handlers, CLI commands, scheduled tasks, event consumers) delegate to this module for execution rather than implementing execution logic themselves
3. No module outside the runtime initiates agent runs, sequences steps, or manages the agent loop
4. The runtime module is the sole owner of execution lifecycle transitions (pending to running, running to completed/failed)
5. Tool invocations and LLM calls during execution originate from within the runtime module's execution path, not from external callers

## Acceptable Implementation Patterns

- A dedicated runtime or engine module that exposes a `run()` or `execute()` entry point, called by all other layers
- A runner class or function that encapsulates the agent loop, instantiated by API handlers and CLI commands but never reimplemented by them
- A message-driven architecture where all execution requests are enqueued and a single consumer module processes them through the agent loop
- A service layer pattern where a single execution service sits between all entry points and the orchestration/integration layers
- A pipeline or chain-of-responsibility pattern where a single coordinator drives step execution regardless of how the run was initiated

## Evidence to Seek

- **Runtime evidence:** Execution traces from different entry points (API, CLI, scheduler) showing the same runtime module handling all runs
- **Trace evidence:** Follow each entry point from request ingress to agent execution — verify they all reach the same runtime module
- **Implementation evidence:** Import analysis showing that only the runtime module imports orchestration and execution internals; other modules import only the runtime's public interface
- **Declarative evidence:** Architecture documentation describing execution centralization (lowest weight — verify against implementation)

## Trace Method

1. Identify all entry points that can trigger an agent execution (search for run/execute/invoke patterns in API handlers, CLI commands, schedulers, message consumers)
2. For each entry point, trace the call chain from the entry point to where the agent loop actually executes
3. Verify that all entry points converge on the same module — the call chains should merge at a single runtime module
4. Inspect the runtime module: confirm it owns the agent loop (step sequencing, completion detection, lifecycle transitions)
5. Search for execution logic outside the runtime module: look for step sequencing, loop constructs over agent steps, or direct tool/LLM invocations in non-runtime code
6. Check import graphs: modules outside the runtime should not import execution internals (step runners, loop managers, execution state machines)

## Verdict Conditions

**PASS when:** A single runtime module is identifiable AND all entry points delegate to it for execution AND no execution logic (step sequencing, agent loop, lifecycle management) exists outside the runtime module. Confirmed by tracing at least two distinct entry points to the same runtime.

**PARTIAL when:** A runtime module exists and most entry points delegate to it, but at least one entry point contains inline execution logic (e.g., a CLI command that reimplements part of the agent loop), OR execution is centralized but some lifecycle transitions happen outside the runtime (e.g., status updates in route handlers).

**GAP when:** No identifiable runtime module exists (execution logic is distributed across handlers, services, and utilities), OR multiple modules independently implement the agent loop, OR entry points contain significant execution logic rather than delegating.

**UNASSESSABLE when:** The codebase contains a runtime module but the entry points are generated, abstracted, or indirected to a degree that prevents tracing the actual execution path from static analysis (e.g., heavy use of dynamic dispatch or plugin registration that cannot be resolved without runtime inspection).

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether dynamically registered entry points (plugin systems, event-driven triggers) actually delegate to the runtime at invocation time. However, tracing the static registration and dispatch mechanism is sufficient for assessment in most cases.

## Boundary Notes

- This principle assesses where execution logic lives, not how it is structured. A system may centralize all execution in one module but have a poor orchestration model (that is Domain 7).
- Helper modules called by the runtime (tool executors, LLM adapters) do not violate P1.1 as long as the runtime module drives the execution flow and these helpers are subordinate to it. The test is ownership of the loop, not whether every line of execution code is in one file.
- In systems with multiple agent types (e.g., a planning agent and an execution agent), each may have its own runtime module. This is acceptable as long as each agent type's execution is centralized within its respective module and no external code reimplements either loop.
- Thin wrappers or adapters around the runtime (e.g., an async wrapper that calls the synchronous runtime) do not violate P1.1 as long as they do not add execution logic.
