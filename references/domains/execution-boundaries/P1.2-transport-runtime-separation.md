## Intent

The HTTP layer must not own execution semantics. Route handlers must accept requests, delegate to the runtime, and return results. Business logic, execution decisions, step sequencing, conditional branching, and direct tool or LLM calls must not live in route handlers.

## Why It Matters

When route handlers contain execution logic, the transport layer becomes the runtime. This creates several production problems. The execution path becomes untestable without a running HTTP server. The same execution logic cannot be reused from other entry points (CLI, scheduler, message queue) without duplicating the route handler's logic. Error handling conflates transport concerns (malformed requests, authentication failures) with execution concerns (tool failures, LLM timeouts). Observability becomes fragmented because execution context is entangled with HTTP request context. Refactoring the API surface requires touching execution logic, and vice versa — every change carries risk across two concerns.

## Applicability

Applies to: All system types with an HTTP API layer
Requires: An HTTP server with route handlers that interact with agent execution

## N/A Conditions

This principle is N/A when the system has no HTTP layer. Examples: a CLI-only tool, a Jupyter notebook agent, a browser-only agent with no backend server, or a system that communicates exclusively via message queues or gRPC streams with no HTTP surface.

## Required Behaviors

1. Route handlers accept and validate incoming requests (parameter extraction, authentication, input validation)
2. Route handlers delegate execution to a runtime module rather than performing execution logic inline
3. Route handlers shape the runtime's response into an HTTP response (status codes, response bodies, headers)
4. No route handler contains agent invocation logic, step sequencing, conditional execution branching, or direct tool/LLM calls
5. Business rules that determine how an agent executes (which tools, which model, what constraints) live in the runtime or configuration layer, not in route handlers

## Acceptable Implementation Patterns

- Route handlers that call a single service or runtime method and return its result, with all execution logic in the service layer
- A controller pattern where route handlers map to thin controller methods that immediately delegate to a domain/service layer
- A command/message pattern where route handlers construct a command object and dispatch it to a handler in the runtime layer
- An async task pattern where route handlers enqueue an execution request and return a task/run identifier, with execution happening in a separate worker that owns the runtime
- A gateway pattern where the HTTP layer is a separate process or module that forwards requests to an execution service via internal API or message passing

## Evidence to Seek

- **Runtime evidence:** Execution traces showing that the same runtime module handles executions regardless of whether they were initiated via HTTP, CLI, or another entry point
- **Trace evidence:** Read 3-5 route handlers end to end — check whether they delegate to a runtime or contain execution logic inline
- **Implementation evidence:** Route handler source code showing the boundary between request handling and execution delegation; import analysis showing route handlers do not import execution internals
- **Declarative evidence:** API documentation or architecture docs describing the transport/runtime separation (lowest weight)

## Trace Method

1. Locate the API route definitions (route files, controller files, handler registrations)
2. Select 3-5 route handlers that interact with agent execution (run creation, step triggers, result retrieval)
3. For each handler, read the full handler body and identify: (a) request handling code (parsing, validation, auth), (b) execution logic (any agent invocation, tool calls, step sequencing, conditional branching), (c) response shaping code (status codes, serialization)
4. Check whether category (b) exists in the handler. If execution logic is present, this is a boundary violation
5. Trace any service/runtime calls made by the handler — verify they lead to the centralized runtime (P1.1), not to ad-hoc execution logic in an intermediate layer
6. Search for execution-related imports in route handler files (agent runners, tool executors, LLM clients imported directly into handler modules)

## Verdict Conditions

**PASS when:** All sampled route handlers delegate execution to a runtime or service layer AND no route handler contains agent invocation logic, step sequencing, conditional execution branching, or direct tool/LLM calls AND the delegation target is consistent across handlers.

**PARTIAL when:** Most route handlers delegate properly but at least one contains inline execution logic (e.g., a webhook handler that processes agent steps directly), OR route handlers delegate but also contain business logic that should live in the runtime (e.g., conditional model selection based on request parameters).

**GAP when:** Route handlers routinely contain execution logic — agent invocations, step sequencing, tool calls, or conditional execution branching happen directly in handler code, OR the route handler layer is the de facto runtime with no separate execution module.

**UNASSESSABLE when:** Route handlers are generated by a framework or decorator system whose behavior cannot be determined from the handler source alone (e.g., a framework that auto-generates execution logic from route annotations, where the actual execution path is not visible in source code).

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether framework middleware injects execution logic into the request pipeline that is not visible in handler source code. However, reading handler code and tracing the delegation chain is sufficient for assessment in standard implementations.

## Related Principles

- [P1.1](P1.1-execution-centralization.md) — P1.2 is a specialization of P1.1 for the HTTP layer. A system can satisfy P1.1 (centralized execution) but violate P1.2 if the centralized runtime is embedded in the route handler layer.
- [P1.4](P1.4-execution-errors-separate.md) — Transport/runtime separation is a prerequisite for clean error separation. When execution logic lives in route handlers, distinguishing execution errors from transport errors becomes architecturally impossible.
- [P1.3](P1.3-ui-as-consumer.md) — If the transport layer owns execution logic, the UI has no clean API to consume. P1.2 violations often force P1.3 violations because the UI must interact with execution logic that should be behind the API.

## Boundary Notes

- This principle assesses the transport layer boundary specifically. Other boundary violations (UI calling runtime directly, schedulers reimplementing the loop) are covered by P1.1 and P1.3.
- Route handlers may contain request-scoped logic that is not execution logic: input validation, authentication checks, rate limiting, request logging, and response serialization are all transport concerns and belong in the handler. The test is whether the handler makes execution decisions, not whether it contains any logic at all.
- A route handler that selects which runtime method to call based on request parameters (e.g., routing to `runtime.run_sync()` vs `runtime.run_stream()` based on an `Accept` header) is performing transport-level dispatch, not execution logic. This is acceptable.
- Middleware that sets up execution context (binding a trace ID, resolving tenant configuration) before the handler runs is transport infrastructure, not execution logic. The test is whether the middleware drives execution or merely configures it.
