## Intent

Execution failures (step failures, tool failures, LLM errors) must not be surfaced as HTTP errors. They belong to the run lifecycle and must be communicated through run state and the event stream. A run that fails during execution must return a success status at the HTTP layer — the outcome is observable via state or events, not via HTTP status codes.

## Why It Matters

When execution failures are surfaced as HTTP 500 errors, clients cannot distinguish between "the server crashed" and "the agent's tool call failed." These are fundamentally different situations requiring different responses. A transport error means the request should be retried. An execution error means the run failed and the client should inspect run state for details. Conflating these categories forces clients into incorrect retry behavior (retrying a deterministically failing execution), prevents meaningful error handling in the UI (showing "server error" when the actual problem is a tool timeout), and corrupts monitoring data (execution failures inflate HTTP error rates, masking real infrastructure problems). Operators lose the ability to distinguish system health from execution success rates.

## Applicability

Applies to: All system types with an HTTP API layer that triggers or reports on agent execution
Requires: An HTTP server that interacts with agent execution lifecycle

## N/A Conditions

This principle is N/A when the system has no HTTP layer. Examples: a CLI-only tool, a Jupyter notebook agent, a browser-only agent with no backend server, or a system that communicates exclusively via message queues or gRPC streams with no HTTP surface.

## Required Behaviors

1. HTTP status codes reflect transport-layer outcomes only: 200/201/202 for successfully received and processed requests, 4xx for client errors (bad input, authentication failure), 5xx only for actual server/infrastructure failures
2. Execution failures (tool errors, LLM failures, step failures, timeout during execution) are represented in the response body as run state, not as HTTP error status codes
3. A run that is accepted and then fails during execution returns HTTP 200/202 with a run state indicating failure, not HTTP 500
4. The response body for execution-related endpoints includes structured run state that distinguishes between successful completion, execution failure, and in-progress states
5. Event streams (SSE, WebSocket) carry execution failure events as typed events within the stream, not as stream termination errors

## Acceptable Implementation Patterns

- REST endpoints that return HTTP 202 (Accepted) with a run identifier, and execution failures are observable by polling the run state endpoint which returns the failure as a state field
- Streaming endpoints that return HTTP 200 with an SSE/event stream, and execution failures are emitted as typed failure events within the stream before the stream closes normally
- Response envelope pattern where all execution-related responses include a status/state field (success, failed, running) alongside the HTTP status code, with HTTP status reflecting only transport success
- Async execution pattern where the API returns immediately with a task reference, and the task's final state (including failure details) is available via a separate status endpoint
- Error payload pattern where execution failures return HTTP 200 with a structured error payload containing failure classification, message, and context

## Evidence to Seek

- **Runtime evidence:** API responses from failed executions showing HTTP 200/202 with failure state in the response body, rather than HTTP 500
- **Trace evidence:** Follow an execution failure (e.g., tool failure, LLM timeout) from the point of failure through the runtime, through the API handler, to the HTTP response — check where the error is classified and what HTTP status is returned
- **Implementation evidence:** API handler error handling code, response shaping logic, error-to-HTTP-status mapping functions or middleware
- **Declarative evidence:** API documentation describing error response formats and status code conventions (lowest weight)

## Trace Method

1. Identify API endpoints that create or interact with agent executions (run creation, step triggers, result retrieval)
2. In the route handler for run creation, trace what happens when the runtime returns an execution failure — does the handler return HTTP 200 with failure state, or does it raise/throw and let error middleware convert it to HTTP 500?
3. Search for error-to-status-code mapping: look for middleware or utility functions that convert exceptions to HTTP responses — check whether execution-related exceptions map to 5xx
4. Check for broad exception handlers in API middleware that catch all exceptions and return HTTP 500 — these often accidentally catch execution failures
5. If the system uses streaming (SSE/WebSocket), examine how execution failures are communicated: as typed events within the stream, or as stream errors/disconnections
6. Sample API response types or schemas for execution endpoints — check whether they include a run state field that can represent failure independently of HTTP status

## Verdict Conditions

**PASS when:** Execution failures are represented as run state in the response body (not as HTTP error status codes) AND HTTP 5xx is reserved for actual transport/infrastructure failures AND streaming execution failures are emitted as typed events within the stream AND this is confirmed by tracing at least one execution failure path from runtime through API handler to HTTP response.

**PARTIAL when:** Some execution failures are correctly surfaced as run state, but others leak as HTTP 500 (e.g., tool failures are handled correctly but LLM timeouts bubble up as HTTP 500), OR execution failures return a non-5xx status code but the response body does not include structured failure state (e.g., returns HTTP 200 with an empty body on failure).

**GAP when:** Execution failures routinely return HTTP 500, OR there is no distinction between transport errors and execution errors in API responses, OR broad exception middleware catches execution failures and converts them to HTTP 5xx without differentiation.

**UNASSESSABLE when:** The error handling path between the runtime and the HTTP response is abstracted by a framework or middleware layer whose behavior cannot be determined from source code (e.g., the framework auto-generates error responses from exceptions using conventions not visible in the codebase).

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** The actual HTTP status codes returned in production for execution failures (may depend on runtime exception handling behavior that differs from what source code suggests). However, tracing the error handling path from the runtime through the API handler to the response shaping code is sufficient for static assessment.

## Boundary Notes

- This principle specifically addresses the mapping between execution outcomes and HTTP status codes. It does not assess the structure of error payloads (that is the error handling domain) or the observability of failures (that is Domain 3).
- HTTP 4xx responses for bad input to execution endpoints (malformed run configuration, invalid tool specification) are transport-layer concerns and are correctly classified as HTTP errors. The principle targets execution failures that occur after the request is accepted and execution begins.
- An API that returns HTTP 500 for genuinely unexpected internal errors (unhandled exceptions in framework code, out-of-memory conditions) is correct — those are infrastructure failures. The violation is returning HTTP 500 for expected execution failures (tool returned an error, LLM rate-limited, step timed out).
- Systems that use HTTP 409 (Conflict) or HTTP 422 (Unprocessable Entity) for execution failures are still violating P1.4 — execution failures belong in run state, not in HTTP status codes, regardless of which specific HTTP error code is used.
- Webhook callbacks that report execution results must also follow this principle: the webhook payload should contain structured run state, and the webhook delivery itself should use HTTP 200 to indicate successful delivery regardless of execution outcome.
