## Intent

When multiple constraints are active with conflicting tool restrictions, the most restrictive intersection must apply. If Skill A allows tools {X, Y, Z} and Skill B allows tools {Y, Z, W}, the composed execution must only have access to {Y, Z}. Permissive composition (union) is a safety violation.

## Why It Matters

Composition is the point where tool access control is most likely to fail silently. Individual skills or workflows may each have well-defined tool restrictions, but when they are combined, the system must decide how to resolve conflicts. A system that takes the union of tool sets effectively grants the most permissive access — the exact opposite of the principle of least privilege. An agent operating under composed constraints that takes the intersection is safe by default; one that takes the union is permissive by default and vulnerable to privilege escalation through composition.

## Applicability

Applies to: Systems where multiple constraints (skills, workflows, policies) can be active simultaneously
Requires: A composition mechanism that combines tool restrictions from multiple sources

## N/A Conditions

N/A when the system has no composition mechanism — single-skill execution with no layering of constraints, or systems where only one set of tool restrictions is ever active at a time. If constraints never overlap, composition semantics are not applicable.

## Required Behaviors

1. When multiple tool restriction sets are active, the system computes the intersection (not the union) of allowed tools
2. A tool that is denied by any active constraint is denied in the composed context, even if another constraint allows it
3. The composition logic is explicit and traceable — not implicit or framework-dependent
4. Composition produces the same result regardless of the order constraints are applied (commutative)
5. The composed tool set is enforced at the runtime level before tool invocation

## Acceptable Implementation Patterns

- Set intersection of tool allowlists when composing multiple skills or workflows
- Policy engine that evaluates all active constraints and applies deny-wins semantics
- Constraint merging layer that explicitly computes the most restrictive tool set
- Layered permission model where each layer can only further restrict (never expand) the tool set from the layer above
- Explicit composition function that takes multiple tool sets and returns their intersection

## Evidence to Seek

- **Runtime evidence:** Compose two contexts with conflicting tool restrictions and verify the intersection is enforced
- **Trace evidence:** Follow the composition path from multiple constraint definitions through merging logic to the final tool set presented to the agent
- **Implementation evidence:** Set intersection or deny-wins logic in the constraint composition code
- **Declarative evidence:** Documentation describing composition semantics (intersection vs. union)

## Trace Method

1. Find where tool restriction sets are defined (per skill, per workflow, per policy)
2. Find the composition or merging logic that combines multiple restriction sets
3. Check whether the merging logic computes intersection (deny-wins) or union (allow-wins)
4. Verify that the composed tool set is what is actually enforced at dispatch time
5. Check for order-dependence: does the result change based on the order constraints are applied?
6. Test mentally with a conflict scenario: Skill A allows tool X, Skill B denies tool X — is X available or denied in the composed context?

## Verdict Conditions

**PASS when:** Composition of conflicting tool restrictions produces the intersection (most restrictive). Deny-wins semantics are applied. The composed tool set is enforced at the runtime level. Composition is order-independent.

**PARTIAL when:** Composition logic exists but is inconsistent. For example: intersection is applied for some constraint types but union for others. Or the composition produces the correct result but is order-dependent. Or intersection is computed but not enforced at dispatch time.

**GAP when:** Composition takes the union of tool sets (most permissive). A tool denied by one constraint is still available because another constraint allows it. Or no composition logic exists — when multiple constraints are active, the system falls back to unrestricted access.

**UNASSESSABLE when:** Composition logic exists but the merging semantics cannot be determined from static analysis (e.g., delegated to a policy engine whose rules are external to the codebase).

## Behavioral Verification Required?

**Required:** Yes
**Recommended test case:** Compose two skills where Skill A allows tool X and Skill B denies tool X. Execute under the composed context and verify that tool X is denied. This test validates that intersection semantics are applied in practice, not just in theory.
**What cannot be verified statically:** Whether the intersection is correctly computed for all possible constraint combinations, and whether edge cases (empty intersection, single constraint, nested composition) are handled correctly. Static analysis can verify the merging logic exists and uses intersection, but behavioral testing confirms it works under realistic composition scenarios.

## Boundary Notes

- P9.2 only applies when multiple constraints are active simultaneously. A system that never composes constraints (always single-skill execution) is N/A for P9.2 but may still have gaps under P9.1.
- The distinction between "deny" and "not explicitly allowed" matters. In an allowlist model, the absence of a tool from an allowlist is equivalent to denial. In a denylist model, tools must be explicitly denied. P9.2 requires that the more restrictive interpretation wins regardless of model.
- P9.2 does not require that composition is impossible or prevented — composition is a valid feature. It requires that composition is safe by default through intersection semantics.
- This principle does not prescribe how constraints are defined (allowlist vs. denylist vs. capability-based); it prescribes how conflicting constraints are resolved (intersection, not union).
