## Intent

For consequential actions, there must be a mechanism to require human approval before execution. The agent runtime must be capable of pausing execution, presenting the pending action to a human, and proceeding only upon approval.

## Why It Matters

Autonomous agents make decisions that can have real-world consequences — writing files, sending messages, executing code, calling external services, modifying data. Some of these actions are irreversible or high-impact. Without a human-in-the-loop mechanism, the system relies entirely on the LLM's judgment for all decisions, including those where the cost of error is high. A human approval gate for consequential actions provides a safety net that bounds the damage an autonomous agent can cause, while still allowing full autonomy for low-risk operations.

## Applicability

Applies to: Systems where the agent can perform consequential actions (writes, mutations, external calls)
Requires: At least one action in the tool catalog that modifies state or has side effects

## N/A Conditions

N/A when all agent actions are read-only. Systems that only retrieve information, generate text, or perform analysis without any side effects do not need a human-in-the-loop mechanism because no action can cause harm.

## Required Behaviors

1. A mechanism exists to designate specific actions or tool invocations as requiring human approval
2. When a designated action is about to execute, the runtime pauses and presents the action to a human for review
3. The action proceeds only upon explicit human approval
4. Rejected actions are handled gracefully — the agent is informed of the rejection and can continue execution with alternative actions
5. The approval mechanism is enforced at the runtime level, not only through prompt instructions

## Acceptable Implementation Patterns

- Confirmation callback in the agent loop that pauses execution and requests approval before designated tool invocations
- Approval queue where consequential actions are queued for human review before execution
- Interactive confirmation in CLI or UI that presents the pending action and waits for user input
- Webhook-based approval where the runtime sends a notification and waits for an external approval signal
- Policy-based approval where certain tool categories or action types automatically trigger the approval flow

## Evidence to Seek

- **Runtime evidence:** Triggering a consequential action and verifying that execution pauses for approval
- **Trace evidence:** Follow a consequential tool invocation from request through the approval check to conditional execution
- **Implementation evidence:** Approval callback or hook in the agent loop, action categorization (consequential vs. safe), approval flow code
- **Declarative evidence:** Configuration that designates which actions require approval, documentation of the approval mechanism

## Trace Method

1. Find the tool dispatch mechanism in the agent loop
2. Check for an approval or confirmation gate before tool execution
3. Identify how actions are categorized (consequential vs. safe) — is it per-tool, per-category, or configurable?
4. Trace the approval flow: what happens when approval is requested? How is the human notified? How does the response reach the runtime?
5. Check the rejection path: what happens when a human denies the action? Does the agent receive feedback?
6. Verify that the approval mechanism is enforced at the runtime level, not just as a prompt instruction

## Verdict Conditions

**PASS when:** A mechanism exists to require human approval for consequential actions. The approval gate is enforced at the runtime level. Designated actions pause execution pending approval. Rejection is handled gracefully.

**PARTIAL when:** An approval mechanism exists but is incomplete. For example: approval exists for some actions but not all consequential ones. Or the mechanism exists but rejection crashes the execution instead of allowing continuation. Or approval is only implemented as a prompt instruction without runtime enforcement.

**GAP when:** No human-in-the-loop mechanism exists. All agent actions execute autonomously without any approval checkpoint, regardless of their consequences.

**UNASSESSABLE when:** An approval mechanism appears to exist (e.g., a confirmation parameter on tool definitions) but the enforcement path cannot be traced through the runtime.

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether the approval mechanism reliably pauses execution in all code paths and whether the UX of the approval flow is sufficient for informed human decision-making. However, verifying the approval gate in the dispatch path and the rejection handling logic is sufficient for static assessment.

## Boundary Notes

- P9.3 assesses whether the capability exists, not whether it is activated for every possible action. A system that has an approval mechanism but only activates it for a subset of consequential actions is PASS — the mechanism exists and can be extended.
- The approval mechanism does not need to be synchronous. Asynchronous approval (queue-based, webhook-based) satisfies P9.3 as long as execution genuinely pauses until approval is received.
- P9.3 applies to the agent runtime, not to the client application. A UI-level confirmation dialog that happens before the agent is invoked does not satisfy P9.3 — the gate must be within the agent's execution loop.
- For fully automated pipelines (CI/CD, batch processing), human-in-the-loop may be impractical. In such cases, compensating controls (pre-approved action lists, dry-run modes, post-execution audit) may partially satisfy the intent.
