## Intent

System prompts must be centralized, versioned, and auditable — not inline strings scattered across source files. Prompts are the primary mechanism through which agent behavior is defined, and they must be governed with the same rigor as configuration or code.

## Why It Matters

System prompts define how the agent reasons, what constraints it follows, what tone it uses, and how it interprets user requests. When prompts are inline strings scattered across source files, they are invisible to governance processes — no one knows which prompt version is active, changes are unreviewed, and drift between environments goes undetected. Inline prompts also resist auditing: finding all prompts in a codebase requires searching for string literals, and distinguishing system prompts from log messages or error strings is unreliable. Centralized, versioned prompts enable review, testing, rollback, and compliance verification.

## Applicability

Applies to: All agent systems that use LLM interactions with system prompts
Requires: At least one system prompt that shapes agent behavior

## N/A Conditions

None — all agentic systems that interact with LLMs use system prompts to define behavior. Even systems with a single prompt benefit from centralization and versioning for auditability.

## Required Behaviors

1. System prompts are stored in dedicated locations (files, templates, registries) rather than inline in source code
2. Prompt changes are trackable — versioned through source control, a prompt registry, or a versioning mechanism
3. All system prompts used by the agent can be enumerated from their storage location without searching through source code
4. Prompt content is loaded at runtime from its governed source, not hard-coded in LLM call sites
5. The mapping between execution contexts and their prompts is explicit and auditable

## Acceptable Implementation Patterns

- Prompt template files (Markdown, Jinja, Mustache) stored in a dedicated directory and loaded at runtime
- Prompt registry with versioned entries and an API for retrieval by context
- Configuration-driven prompt selection where prompt file paths or registry keys are specified in configuration
- Prompt management system with version history, A/B testing, and rollback capability
- Separate prompt module or package that exports all system prompts as named, versioned artifacts

## Evidence to Seek

- **Runtime evidence:** Changing a prompt in its governed location and verifying the agent behavior changes without code modification
- **Trace evidence:** Follow a system prompt from its LLM call site back to its source — is it loaded from a file/registry or hard-coded inline?
- **Implementation evidence:** Prompt file directory, prompt loading code, prompt versioning mechanism, absence of inline prompt strings at LLM call sites
- **Declarative evidence:** Prompt inventory documentation, prompt management guidelines

## Trace Method

1. Find all LLM call sites in the codebase (API calls to model providers or abstraction layers)
2. For each call site, identify the system prompt source: inline string, variable reference, file load, or registry lookup
3. Check whether prompts are centralized in a dedicated location or scattered across modules
4. Verify that prompt content can be enumerated from the storage location without code search
5. Check for versioning: are prompt changes tracked through source control or a versioning mechanism?
6. Check the mapping: is it clear which prompt is used in which execution context?

## Verdict Conditions

**PASS when:** System prompts are centralized in governed locations (files, registries). Prompts are loaded at runtime from these locations. Changes are versioned and trackable. All prompts can be enumerated from their storage location.

**PARTIAL when:** Some prompts are centralized but others are inline. Or prompts are in separate files but are not versioned or the mapping between contexts and prompts is unclear. Or prompts are centralized but loaded through a mechanism that mixes prompt content with other string construction.

**GAP when:** System prompts are inline strings in source files. No centralized prompt storage exists. Prompts cannot be enumerated without searching through source code. Changes to prompts require modifying source code at LLM call sites.

**UNASSESSABLE when:** Prompts are loaded from an external source (prompt management service, database) whose content and versioning cannot be inspected from the codebase.

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether the governed prompts are the actual prompts used at runtime (runtime could override or modify them). However, verifying the loading mechanism and absence of inline prompts is sufficient for static assessment.

## Boundary Notes

- P9.4 assesses whether prompts are governed (centralized, versioned, auditable); it does not assess prompt quality, effectiveness, or safety content
- Source control versioning of prompt files satisfies the versioning requirement — a dedicated prompt versioning system is not required
- Dynamic prompt construction (assembling prompts from templates with runtime variables) is acceptable as long as the templates are governed. The template source must be centralized; the variable injection can happen at runtime.
- P9.4 applies to system prompts (prompts that define agent behavior), not to user messages or tool outputs that flow through the agent
