## Intent

The agent must not have unrestricted access to all tools in all contexts. Tool access control must exist — a mechanism to scope, restrict, or filter which tools are available to the agent based on context, workflow, skill, user, or execution mode.

## Why It Matters

An agent with unrestricted tool access is an agent where every execution can invoke any capability the system offers. This is the autonomous-agent equivalent of running all code as root. When a summarization workflow can invoke code execution tools, when a read-only analysis skill can invoke write tools, or when an untrusted user's agent session has access to administrative tools, the system has a tool access control gap. The consequences compound with agent autonomy — an agent that reasons about which tools to use will use whatever is available, and unrestricted availability means unrestricted potential for unintended actions.

## Applicability

Applies to: All agent systems that invoke tools
Requires: A tool catalog with more than one tool

## N/A Conditions

None — any agentic system with tools should have a mechanism to control which tools are accessible in which contexts. Even single-user systems benefit from context-scoped tool availability to prevent unintended actions.

## Required Behaviors

1. A mechanism exists to define which tools are accessible in a given context (workflow, skill, user, execution mode)
2. The agent loop respects tool access restrictions — restricted tools are not presented to the LLM and cannot be invoked
3. Tool access control is enforced at the runtime level, not only at the prompt level (telling the LLM not to use a tool is not access control)
4. Tool access restrictions can be defined declaratively (configuration, definition files) rather than requiring code changes
5. Default tool access is restrictive rather than permissive (tools must be explicitly granted, not explicitly denied)

## Acceptable Implementation Patterns

- Tool allowlists per workflow or skill that restrict which tools are available during execution
- Role-based tool access where user roles map to tool sets
- Context-aware tool filtering where the runtime filters the tool catalog before presenting it to the LLM
- Tool capability scoping where tools are grouped by capability level (read-only, write, admin) and contexts are assigned capability levels
- Policy-based tool access where a policy engine evaluates tool access at invocation time

## Evidence to Seek

- **Runtime evidence:** Executing in a restricted context and verifying that restricted tools are not available to the agent
- **Trace evidence:** Follow a tool invocation from request through access control check to dispatch, confirming the check exists and is enforced
- **Implementation evidence:** Tool filtering or access control code, tool allowlist definitions, access check in the dispatch path
- **Declarative evidence:** Tool access policy definitions, skill-level tool restrictions, documentation of access control model

## Trace Method

1. Find the point where tools are made available to the agent (tool list construction, LLM prompt assembly)
2. Check whether the tool list is filtered or scoped based on context (workflow, skill, user)
3. Find the tool dispatch mechanism and check for access control checks before invocation
4. Search for tool allowlists or denylists in configuration or definition files
5. Check whether the default behavior is permissive (all tools available unless restricted) or restrictive (no tools available unless granted)
6. Verify that access control is enforced at the runtime level, not only through prompt instructions

## Verdict Conditions

**PASS when:** Tool access is scoped by context. A mechanism exists to restrict which tools are available per workflow, skill, or execution context. Restrictions are enforced at the runtime level (not just prompt-level). Default access is restrictive.

**PARTIAL when:** Tool access control exists but is incomplete. For example: some contexts restrict tools but others have full access. Or access control exists but default access is permissive (all tools available unless explicitly denied). Or restrictions are enforced only at the prompt level without runtime enforcement.

**GAP when:** No tool access control exists. All tools are available in all contexts. The agent has unrestricted access to the full tool catalog regardless of workflow, skill, or user.

**UNASSESSABLE when:** Tool filtering logic exists but the filtering criteria cannot be determined from static analysis (e.g., access decisions are made by an external policy service whose rules are not in the codebase).

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether restricted tools are truly inaccessible at runtime (the LLM could potentially reference tool names even if they are not in the tool list). However, verifying the filtering mechanism and dispatch-level enforcement is sufficient for static assessment.

## Boundary Notes

- P9.1 assesses whether tool access control exists; P9.2 assesses how conflicting restrictions are resolved under composition. A system can pass P9.1 (tool access is scoped) while failing P9.2 (composition takes the union instead of the intersection).
- Prompt-level tool restriction (instructing the LLM not to use certain tools) is not sufficient for P9.1. The LLM is not a reliable enforcement mechanism — it may ignore instructions. Runtime-level enforcement is required.
- User authentication and authorization (who can access the system) is a separate concern from tool access control (which tools are available in which agent context). P9.1 focuses on the latter.
- A system that has only one tool is still subject to P9.1 if different contexts should have different access to that tool (e.g., read-only mode should not have access to a write tool).
