## Intent

When autonomous execution hits a guardrail (step limit, token limit, budget limit, time limit), the run must complete with a truncation indicator — not fail with an exception. The system must distinguish between "the agent completed its work" and "the agent was stopped by a guardrail" without treating guardrail enforcement as a system error.

## Why It Matters

Guardrails exist to bound autonomous execution. They are a safety mechanism, not an error condition. When hitting a guardrail produces a failure (exception, crash, error status), the system conflates safety enforcement with system malfunction. Operators cannot distinguish between a run that failed due to a bug and a run that was intentionally bounded. Monitoring systems generate false alerts. Retry logic may re-execute a run that was correctly bounded, consuming additional resources. Treating guardrail hits as truncation (completed with indicator) preserves the semantic distinction between "stopped safely" and "failed unexpectedly."

## Applicability

Applies to: Systems with guardrails that bound autonomous execution (step limits, token limits, budget limits, time limits)
Requires: At least one guardrail mechanism that can interrupt or terminate execution

## N/A Conditions

N/A when the system has no guardrails — no step limits, no token limits, no budget limits, no time limits. Note: the absence of guardrails is itself a gap under P7.3 (Guardrail Enforcement). A system that is N/A for P9.5 due to missing guardrails likely has a separate safety gap.

## Required Behaviors

1. When a guardrail limit is reached, execution completes with a defined completion status (not an exception or error)
2. The completion status distinguishes between normal completion and guardrail-triggered truncation
3. The truncation indicator is part of the run's result — visible to callers, logged in events, and persisted in run state
4. Guardrail truncation does not trigger error handling, retry logic, or failure alerting
5. Partial results produced before truncation are preserved and returned (not discarded)

## Acceptable Implementation Patterns

- Completion status enum with distinct values for COMPLETED and TRUNCATED (or equivalent: COMPLETED_WITH_LIMIT, BOUNDED)
- Run result object that includes a completion reason field indicating which guardrail was triggered
- Event emission at truncation with a specific event type (e.g., run_truncated) distinct from failure events
- Graceful shutdown sequence when a limit is reached: finalize current step, persist partial results, set truncation status, return
- Budget/token tracking that transitions to a truncation state rather than raising an exception when exhausted

## Evidence to Seek

- **Runtime evidence:** Triggering a guardrail limit and verifying the run completes with truncation status, not failure
- **Trace evidence:** Follow the guardrail check code path when the limit is exceeded — does it raise an exception or set a completion status?
- **Implementation evidence:** Guardrail limit checks, truncation status values, completion reason fields, absence of exceptions on limit hit
- **Declarative evidence:** Documentation describing guardrail behavior, completion status definitions

## Trace Method

1. Find all guardrail limit checks in the codebase (max_steps, max_tokens, budget_limit, timeout)
2. For each limit check, trace the code path when the limit is exceeded
3. Check whether the limit path raises an exception, returns an error, or sets a completion status
4. If a completion status is set, check whether it distinguishes between normal completion and truncation
5. Check whether partial results are preserved on truncation or discarded
6. Check whether the truncation status is included in events, logs, and API responses

## Verdict Conditions

**PASS when:** Guardrail hits produce a truncation completion status (not failure). The status distinguishes truncation from normal completion. Partial results are preserved. The truncation indicator is visible in run state, events, and API responses.

**PARTIAL when:** Some guardrails produce truncation status but others raise exceptions. Or truncation status exists but partial results are discarded. Or truncation is handled correctly internally but the API response does not distinguish truncation from normal completion.

**GAP when:** Guardrail hits raise exceptions or produce error status. The system treats hitting a guardrail the same as a system failure. No distinction exists between truncated and failed runs.

**UNASSESSABLE when:** Guardrail checks exist but the code path on limit exceeded cannot be fully traced (e.g., delegated to a framework's internal handling that is not visible in the codebase).

## Behavioral Verification Required?

**Required:** Yes
**Recommended test case:** Configure a guardrail with a low limit (e.g., max_steps = 2 for a task that requires more steps). Execute the task and verify: (1) the run completes with a truncation or bounded status, not a failure or error status; (2) partial results from completed steps are preserved in the response; (3) the completion reason indicates which guardrail was triggered. This test validates that guardrail enforcement produces graceful truncation in practice, not just in the code's apparent logic.
**What cannot be verified statically:** Whether all guardrail code paths genuinely produce truncation at runtime. Edge cases (concurrent limit hits, limits reached mid-tool-execution, limits reached during LLM streaming) may behave differently from the primary code path. Behavioral testing confirms the end-to-end behavior under realistic conditions.

## Boundary Notes

- P9.5 applies only to guardrail-triggered limits, not to genuine system errors. A tool that throws an exception due to a bug is a failure, not a truncation — P9.5 does not apply to that scenario.
- The specific completion status values (TRUNCATED, BOUNDED, COMPLETED_WITH_LIMIT) are implementation choices. What matters is that the status is semantically distinct from both normal completion and failure.
- A system that has no guardrails is N/A for P9.5 but likely has a gap under P7.3. If P7.3 is also N/A (no guardrail concept), then P9.5 is genuinely not applicable. If P7.3 is a GAP (guardrails should exist but do not), then P9.5 is also effectively a GAP because the missing guardrails would need graceful enforcement.
- Timeout-based guardrails (hard time limits) are the most likely to produce exceptions rather than truncation, because timeout mechanisms in most runtimes are interrupt-based. Special attention should be given to timeout handling paths.
