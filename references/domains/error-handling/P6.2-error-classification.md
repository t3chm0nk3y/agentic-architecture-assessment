## Intent

Errors must be classified by type: validation failure, tool failure, LLM failure, timeout, internal error, and other categories relevant to the system. Classification enables the system and its operators to respond differently to different failure modes.

## Why It Matters

Not all errors are equal. A validation error means the input is wrong and retrying will fail again. A timeout may be transient and retriable. An LLM failure may require fallback to a different model or a reduced context. An internal error indicates a bug. Without classification, every error triggers the same handling path — either everything is retried (wasteful and potentially harmful) or nothing is retried (brittle). Classification is the foundation for intelligent error handling, operator dashboards, and automated remediation.

## Applicability

Applies to: All system types
Requires: Multiple failure modes that warrant different handling (nearly universal in agentic systems)

## N/A Conditions

N/A when the system has genuinely only one failure mode and one handling path. This is extremely rare in agentic systems due to the inherent diversity of failure sources (LLM, tools, infrastructure, validation).

## Required Behaviors

1. Errors carry a classification field that identifies the category of failure
2. The classification scheme covers the major failure sources in the system (at minimum: validation, tool/integration, LLM/model, timeout, internal/unexpected)
3. The classification is machine-readable (enum, constant, code) rather than embedded in the error message text
4. Error handling logic uses the classification to determine behavior (different handling for different error types)
5. The classification scheme is documented or self-documenting (enum names, constant definitions)

## Acceptable Implementation Patterns

- Enum or string constant set defining error categories, referenced by error model and handling logic
- Typed exception hierarchy where each exception subclass represents a classification (ValidationError, ToolError, LLMError, TimeoutError)
- Error code scheme with documented categories (4xx for validation, 5xx for internal, custom ranges for domain-specific)
- Tagged union / algebraic data type with variant per error class
- Error type field on a flat error struct with constrained values

## Evidence to Seek

- **Runtime evidence:** Error output showing classification fields with distinct values for different failure types
- **Trace evidence:** Follow two different failure paths (e.g., a validation error and a tool error) and verify they produce differently classified errors that are handled differently
- **Implementation evidence:** Error classification enum/constants, error type field definitions, conditional handling based on error type
- **Declarative evidence:** Error classification documentation or error code tables

## Trace Method

1. Find the error model (P6.1) and identify the classification field
2. Enumerate the classification values — what categories exist?
3. Check coverage: are the major failure sources in the system represented? (LLM calls, tool invocations, input validation, infrastructure, internal errors)
4. Find error handling code (catch blocks, error middleware, result matching) and check whether handling branches on the classification
5. Look for catch-all handlers that treat all errors identically — do they exist alongside classification-aware handlers or are they the only handling path?
6. Check whether classification is assigned at error creation time or retroactively (creation-time is preferred)

## Verdict Conditions

**PASS when:** Errors carry a machine-readable classification field AND the classification scheme covers the system's major failure sources AND error handling logic branches on classification to provide different behavior for different error types.

**PARTIAL when:** A classification scheme exists but is incomplete (e.g., only distinguishes "expected" from "unexpected" without finer granularity), OR classification exists but handling logic does not branch on it (classification is recorded but not acted upon).

**GAP when:** No error classification exists — all errors are treated as a single undifferentiated type, OR classification is only present in human-readable error messages and not in a machine-readable field.

**UNASSESSABLE when:** Error types exist with different names but whether they are used for classification-based handling cannot be determined from static analysis (e.g., all caught by a single generic handler at a higher level).

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether the classification values assigned at runtime accurately reflect the actual failure cause. However, the classification scheme structure, assignment patterns, and handling branching are sufficient for static assessment.

## Related Principles

- [P6.1](P6.1-structured-error-model.md) — The error model must carry a classification field. If P6.1 is a GAP (no structured model), P6.2 is necessarily also a GAP.
- [P6.5](P6.5-retry-safety.md) — Retry logic should use error classification to determine retriability. Retrying a validation error is wasteful; retrying a timeout may be appropriate.
- [P6.3](P6.3-failure-propagation.md) — Classification must survive propagation. If errors lose their type during propagation, classification at the creation site provides no value at the handling site.

## Boundary Notes

- P6.2 assesses whether errors are classified, not whether the classification scheme is complete or ideal. A system that distinguishes five error types is better than one that distinguishes none, even if the taxonomy is imperfect.
- Error classification is distinct from HTTP status codes or API response codes, though these may encode classification. The principle assesses internal error handling classification, not just API-facing error responses.
- A system may have implicit classification through exception hierarchy (catching ToolError separately from ValidationError) without an explicit enum — this satisfies P6.2 as long as handling branches on the classification.
