## Intent

Step or component failures must propagate with context to the run level. Silent swallowing of errors that leave the run in an indeterminate state is a gap.

## Why It Matters

An agentic run consists of multiple steps, tool invocations, and LLM calls. When any of these fail, the failure must reach the run-level outcome so that the operator, caller, or orchestrator knows the run did not complete successfully. Silent error swallowing — where a step fails but the run continues as if nothing happened or terminates without recording the failure — is one of the most dangerous patterns in agentic systems. It produces runs that appear successful but produced incorrect or incomplete results, eroding trust and making debugging nearly impossible.

## Applicability

Applies to: All system types with multi-step execution
Requires: Any execution model where a failure in a sub-component should affect the run-level outcome

## N/A Conditions

None — this principle applies to all agentic systems. Even single-step executions must propagate tool or LLM failures to the caller.

## Required Behaviors

1. Failures in steps, tool invocations, or LLM calls propagate to the run-level outcome rather than being silently swallowed
2. Propagated failures retain their original context: error classification, diagnostic detail, and originating component/step
3. The run-level outcome reflects that a failure occurred — the run is not marked as successful when a component failure affected the result
4. Error context is enriched during propagation (each layer adds its own context) rather than stripped (each layer reduces information)
5. No critical code path contains silent error swallowing (bare `except: pass`, empty catch blocks, or errors caught and discarded without recording)

## Acceptable Implementation Patterns

- Exception propagation with context wrapping at each layer boundary (lower-level exception wrapped in higher-level exception with additional context)
- Result type propagation where error variants flow upward through the call chain with context accumulation
- Event-based error propagation where failures emit structured error events consumed by the run-level orchestrator
- Error accumulation pattern where non-fatal step errors are collected and attached to the run outcome as a list of partial failures
- Central error handler at the run level that catches all unhandled exceptions and records them as the run outcome

## Evidence to Seek

- **Runtime evidence:** A failed run whose outcome record includes the originating failure with full context (which step, what failed, why)
- **Trace evidence:** Follow a failure from a tool call or LLM invocation through intermediate handlers to the run-level outcome — check that context survives and the run is marked as failed
- **Implementation evidence:** Exception handling chains, result type propagation, error event emission, run outcome recording code
- **Declarative evidence:** Error handling flow documentation

## Trace Method

1. Find a tool invocation or LLM call in the execution path
2. Identify what happens when that call fails (exception, error result, error code)
3. Trace the failure upward: does it propagate to the step level? Does the step propagate to the run level?
4. At each propagation boundary, check: is context preserved? Is context enriched? Or is context stripped or lost?
5. Check whether the run-level outcome reflects the failure — is the run marked as failed/errored when a component fails?
6. Search for silent error swallowing patterns: bare `except: pass`, empty catch blocks, errors caught with only a log statement and no propagation
7. Check edge cases: what happens when a non-critical step fails? Is it recorded even if the run continues?

## Verdict Conditions

**PASS when:** Failures propagate from components/steps to the run level with context preserved AND the run outcome reflects component failures AND no silent error swallowing is found in critical code paths.

**PARTIAL when:** Failures propagate but lose context during propagation (e.g., original error classification or detail is stripped), OR failures propagate for some components but not others, OR non-critical failures are silently swallowed without recording.

**GAP when:** Failures are silently swallowed in critical code paths — components fail but the run continues as if nothing happened or terminates without recording the failure, OR the run outcome does not reflect that a component failure occurred.

**UNASSESSABLE when:** Error handling code exists at multiple layers but the propagation path between layers cannot be determined from static analysis (e.g., error handling delegated to a framework not visible in the codebase).

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether propagation actually works correctly under all failure conditions at runtime. However, tracing the propagation path through code, checking for error swallowing patterns, and verifying run outcome recording is sufficient for static assessment.

## Related Principles

- [P6.1](P6.1-structured-error-model.md) — Propagation preserves or loses error structure. If the error model is unstructured (P6.1 GAP), propagation has nothing structured to preserve.
- [P3.5](../observability/P3.5-failure-contextualization.md) — P3.5 assesses whether failures are observable with context; P6.3 assesses whether failures reach the level where they can be observed. A system can have P6.3 PASS (failures propagate) but P3.5 GAP (propagated failures are not logged or surfaced).
- [P7.2](../orchestration-model/P7.2-explicit-state-transitions.md) — State transitions depend on failure propagation. If failures do not propagate, the orchestrator cannot transition to an error state.

## Boundary Notes

- P6.3 is specifically about propagation to the run level. A step that handles an error internally and produces a degraded but valid result is not a propagation failure — the step handled the error. P6.3 is concerned with errors that should affect the run outcome but are silently discarded.
- Intentional error suppression in non-critical paths (e.g., best-effort telemetry, optional cache writes) may be acceptable if the suppression is deliberate, documented or logged at warning level, and does not affect execution correctness. The concern is unintentional or careless suppression in paths that affect results.
- Error accumulation (collecting multiple step failures and attaching them to the run outcome) is a valid propagation pattern — not every failure needs to halt execution immediately, but every failure must be recorded.
