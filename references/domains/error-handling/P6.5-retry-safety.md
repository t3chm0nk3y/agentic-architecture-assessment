## Intent

Where retries are implemented, they must be safe: idempotent operations only, bounded retry counts, and exponential backoff. Retry logic that is unbounded, applied to non-idempotent operations, or lacking backoff amplifies transient failures into sustained outages.

## Why It Matters

Retry logic is a common resilience pattern, but unsafe retries cause more damage than no retries at all. Unbounded retries against a failing service create sustained load that prevents recovery. Retries on non-idempotent operations (e.g., retrying a tool call that creates a resource) produce duplicate side effects. Retries without backoff amplify thundering herd problems. In agentic systems, where tool calls may have real-world side effects (sending emails, creating records, executing code), retry safety is directly tied to operational safety.

## Applicability

Applies to: Systems that implement retry logic for any operation (LLM calls, tool invocations, infrastructure calls)
Requires: At least one code path that retries a failed operation

## N/A Conditions

N/A when the system implements no retry logic whatsoever — all failures are immediately propagated without retry. Note: absence of retry logic is not itself a gap (the system may be designed for fail-fast behavior), but if retries are present, they must be safe.

## Required Behaviors

1. Retry counts are bounded with a defined maximum number of attempts
2. Retries use backoff (exponential, linear, or equivalent) rather than immediate retry at full rate
3. Retries are applied only to operations that are safe to retry (idempotent operations, or operations with idempotency mechanisms like idempotency keys)
4. Non-retriable errors (validation failures, authentication errors, permanent failures) are not retried — error classification (P6.2) informs retriability
5. The system's behavior after exhausting retries is defined: propagate the final error, degrade gracefully, or invoke a fallback — not loop indefinitely

## Acceptable Implementation Patterns

- Retry decorator or wrapper with configurable max_retries, backoff_factor, and retriable_exceptions
- Exponential backoff with jitter to prevent thundering herd synchronization
- Idempotency key generation for non-naturally-idempotent operations that must be retried
- Retry policy objects that encapsulate retry configuration and retriability rules per operation type
- Circuit breaker combined with retry: retry within the circuit breaker's closed state, stop retrying when the circuit opens

## Evidence to Seek

- **Runtime evidence:** Logs showing retry attempts with increasing delays, eventually stopping at a maximum count
- **Trace evidence:** Follow a retry loop from first attempt through retries to final outcome — check for bounds, backoff, and outcome after exhaustion
- **Implementation evidence:** Retry configuration (max retries, backoff settings), retry loop code, retriability checks based on error type, idempotency mechanisms
- **Declarative evidence:** Retry policy documentation or configuration files

## Trace Method

1. Search for retry patterns: retry decorators, retry loops (`while`, `for` with retry counters), retry libraries or utilities
2. For each retry site, check the bound: is there a maximum retry count? What is it?
3. Check for backoff: is there a delay between retries? Does the delay increase (exponential, linear)?
4. Check retriability determination: does the retry logic check error type before retrying? Are non-retriable errors excluded?
5. Check what happens after retry exhaustion: is the error propagated? Is there a fallback? Or does the system enter an undefined state?
6. Identify the operations being retried: are they idempotent? If not, is there an idempotency mechanism (idempotency keys, deduplication)?
7. Look for unbounded retry patterns: `while True` retry loops, retries without a counter, or retries with an unreasonably high bound

## Verdict Conditions

**PASS when:** All retry logic in the system is bounded AND uses backoff AND is applied to idempotent or idempotency-protected operations AND non-retriable errors are excluded AND behavior after retry exhaustion is defined.

**PARTIAL when:** Retry logic exists and is mostly safe but has gaps: some retry sites lack backoff, OR retriability is not checked (all errors retried), OR one retry site is unbounded while others are bounded.

**GAP when:** Retry logic is unbounded (no maximum retry count, infinite retry loops) OR retries are applied to non-idempotent operations without idempotency protection OR retries lack any form of backoff.

**UNASSESSABLE when:** Retry logic is delegated to an external library or framework whose configuration cannot be determined from static analysis.

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether retry backoff timing is correct in practice, or whether operations classified as idempotent are truly idempotent at runtime. However, the retry configuration (bounds, backoff, retriability checks) and the nature of retried operations are sufficient for static assessment.

## Boundary Notes

- Severity is context-dependent: MEDIUM when unbounded retries are found (risk of amplification), LOW when retry logic is simply absent (no retry is safer than unsafe retry). The absence of retry logic is only a concern if the system demonstrably needs resilience against transient failures and has none.
- P6.5 assesses the safety of retry mechanisms, not whether retry logic is necessary. A system that deliberately uses fail-fast behavior with no retries is not a GAP — it is a design choice that may be appropriate.
- Retry logic in test code or development utilities is out of scope. P6.5 assesses retry logic in production/operational code paths.
- Jitter in backoff is a best practice but not a hard requirement. Exponential backoff without jitter is PASS; immediate retry without any delay is GAP.
