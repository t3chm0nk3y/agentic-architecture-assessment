## Intent

Errors must be first-class entities with consistent structure: error class/type, human-readable message, structured diagnostic detail, and execution context.

## Why It Matters

When errors are represented as bare strings or unstructured exceptions, every consumer of error information must parse, guess, or ignore the error. Operators cannot filter errors by type in dashboards. Retry logic cannot distinguish between retriable and terminal failures. Audit logs contain only "something went wrong" instead of what, where, and why. A structured error model ensures that every error carries the information needed for automated handling and human diagnosis.

## Applicability

Applies to: All system types
Requires: Any code path that produces, handles, or reports errors

## N/A Conditions

None — this principle applies to all agentic systems. Every system that can fail must model its errors as structured entities.

## Required Behaviors

1. Errors are represented as typed entities (classes, structs, tagged unions, or equivalent) rather than bare strings or generic exceptions
2. Each error entity carries a classification or type field that identifies the category of failure
3. Each error entity carries a human-readable message suitable for logging and operator display
4. Each error entity carries a structured diagnostic detail field containing context beyond the message (input data, relevant state, upstream error information)
5. Each error entity carries or is associated with execution context (run identifier, step identifier, component name)
6. The error model is used consistently across modules — not just in one component while others use bare strings

## Acceptable Implementation Patterns

- Typed exception hierarchy with classification enum, message, detail dict, and context fields
- Result/Either types with structured error variants carrying classification and detail
- Error builder pattern that enforces required fields at construction time
- Flat error struct/class with required fields validated at creation
- Error wrapping pattern where lower-level errors are wrapped with additional context at each layer

## Evidence to Seek

- **Runtime evidence:** Error output from a failed execution showing structured fields (type, message, detail, context) rather than bare messages
- **Trace evidence:** Follow an error from creation through propagation to final output — check that structured fields survive the journey
- **Implementation evidence:** Error model definitions (classes, types, schemas), error construction sites, error field access patterns
- **Declarative evidence:** Error model documentation or API error response schemas

## Trace Method

1. Search for error model definitions: custom exception classes, error types, error schemas, or error factory functions
2. Inspect the error model for required fields: classification/type, message, diagnostic detail, execution context
3. Find error construction sites — where are errors created? Check that they populate all structured fields
4. Find error consumption sites — where are errors caught, logged, returned, or displayed? Check that they access structured fields rather than just calling `str()` or `.message`
5. Check consistency: are the same error types used across modules, or does each module define its own ad-hoc error representation?
6. Look for bare string errors or generic exceptions in critical code paths that bypass the structured model

## Verdict Conditions

**PASS when:** A structured error model exists with classification, message, diagnostic detail, and context fields AND is used consistently across critical code paths AND errors are constructed with populated fields at creation sites.

**PARTIAL when:** A structured error model exists but is incomplete (e.g., classification without diagnostic detail, or detail without context), OR the model exists but some modules bypass it and use bare strings or generic exceptions.

**GAP when:** No structured error model exists — errors are represented as bare strings, generic exceptions with only a message, or untyped dictionaries with inconsistent fields.

**UNASSESSABLE when:** Error types exist but their runtime behavior (which fields are populated, whether detail is meaningful) cannot be determined from static analysis alone.

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether error detail fields contain meaningful diagnostic information at runtime versus empty or placeholder values. However, the model structure, construction patterns, and field population at creation sites are sufficient for static assessment.

## Related Principles

- [P3.5](../observability/P3.5-failure-contextualization.md) — P3.5 assesses whether failures are observable with context; P6.1 assesses whether the error model itself is structured. A system can have P6.1 PASS (structured model) but P3.5 GAP (errors not logged or surfaced).
- [P6.2](P6.2-error-classification.md) — Error classification (P6.2) depends on the error model carrying a classification field (P6.1). If P6.1 is a GAP, P6.2 is necessarily also a GAP.
- [P6.3](P6.3-failure-propagation.md) — Failure propagation preserves or loses error structure. A structured model (P6.1) that is discarded during propagation (P6.3 GAP) provides no downstream value.

## Boundary Notes

- P6.1 assesses the error model as a data structure. Whether errors are logged, surfaced, or observable is P3.5. Whether they propagate is P6.3. P6.1 is specifically about: do errors carry structured information?
- A single "Error" class with only a message string does not satisfy P6.1 — the model must carry classification, detail, and context beyond just a message.
- Language-native error mechanisms (Python exceptions, Go errors, Rust Results) are acceptable foundations as long as they are extended with the required structured fields. Using `Exception("something broke")` without subclassing or adding context does not satisfy P6.1.
