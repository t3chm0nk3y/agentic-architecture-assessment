## Intent

Non-critical infrastructure failures must not crash the system. Graceful degradation is required: when a non-essential dependency fails, the system must continue operating with reduced functionality rather than terminating entirely.

## Why It Matters

Agentic systems often depend on multiple infrastructure components beyond the core execution path: caches, metrics collectors, logging services, configuration stores, notification systems, and external APIs. When any of these non-critical dependencies fail, the system should degrade gracefully — not crash. A metrics endpoint timeout should not abort an agent run. A cache miss should not prevent tool execution. An unavailable logging service should not block the critical path. Systems that crash on non-critical infrastructure failures have availability proportional to their least reliable dependency.

## Applicability

Applies to: Systems with external infrastructure dependencies beyond the core execution path
Requires: At least one non-critical external dependency (cache, metrics, logging service, notification system, etc.)

## N/A Conditions

N/A when the system has no external infrastructure dependencies — a self-contained single-process agent with no cache, no external metrics, no external logging, and no non-essential service calls. This is rare but possible for simple agent implementations.

## Required Behaviors

1. Non-critical infrastructure failures are caught and handled without crashing the system or aborting the current execution
2. The system distinguishes between critical dependencies (whose failure should stop execution) and non-critical dependencies (whose failure should trigger degradation)
3. Degradation behavior is defined for each non-critical dependency: what happens when the cache is unavailable, when metrics cannot be sent, when the logging service is down
4. Infrastructure failures are recorded (logged or evented) even when they do not stop execution — silent degradation without any signal is a partial gap
5. Degradation does not silently corrupt execution results — the system either produces correct results with reduced functionality or signals that results may be incomplete

## Acceptable Implementation Patterns

- Try/catch blocks around non-critical infrastructure calls with fallback behavior (cache miss falls back to direct computation, metrics failure falls back to no-op)
- Circuit breaker pattern that detects repeated infrastructure failures and stops attempting calls for a cooldown period
- Fallback chain pattern where the system tries a primary dependency, then a secondary, then a local fallback
- Feature flag / capability detection where the system checks dependency availability at startup and disables features that depend on unavailable infrastructure
- Timeout-bounded infrastructure calls with defined behavior on timeout (proceed without the result, use a default)

## Evidence to Seek

- **Runtime evidence:** System continues operating when a non-critical dependency is unavailable, with logged degradation messages
- **Trace evidence:** Follow a non-critical infrastructure call (cache read, metrics push, notification send) and check what happens when it fails — does the system crash, degrade, or silently continue?
- **Implementation evidence:** Try/catch blocks around infrastructure calls, fallback logic, circuit breaker configuration, timeout settings on non-critical calls
- **Declarative evidence:** Architecture documentation identifying critical vs. non-critical dependencies and degradation behavior

## Trace Method

1. Identify all external infrastructure dependencies (databases, caches, message queues, metrics services, logging services, notification systems, external APIs)
2. Classify each dependency: is it critical (execution cannot proceed without it) or non-critical (execution can proceed with degraded functionality)?
3. For each non-critical dependency, find the call sites in code
4. Check what happens when the call fails: is the failure caught? Is there fallback behavior? Does the system crash?
5. Check whether infrastructure failures are recorded (logged, evented) during degradation
6. Look for unprotected infrastructure calls — calls to non-critical services that are not wrapped in error handling

## Verdict Conditions

**PASS when:** Non-critical infrastructure failures are caught and handled with defined degradation behavior AND failures are recorded AND degradation does not corrupt execution results.

**PARTIAL when:** Some non-critical infrastructure failures are handled but others are not (inconsistent resilience), OR degradation occurs but is not recorded (silent degradation), OR the distinction between critical and non-critical dependencies is unclear.

**GAP when:** Non-critical infrastructure failures crash the system or abort execution — no graceful degradation exists for non-essential dependencies.

**UNASSESSABLE when:** Infrastructure dependencies exist but the failure handling behavior cannot be determined from static analysis (e.g., handled by middleware, proxy, or infrastructure layer not visible in the codebase).

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether degradation actually works correctly under real infrastructure failure conditions. However, the presence of error handling around infrastructure calls, fallback logic, and timeout configuration is sufficient for static assessment.

## Related Principles

- [P6.3](P6.3-failure-propagation.md) — Infrastructure failure resilience involves catching errors that should NOT propagate to the run level. P6.3 assesses that critical failures DO propagate; P6.4 assesses that non-critical failures are handled gracefully without propagating fatally.
- [P6.5](P6.5-retry-safety.md) — Infrastructure calls may be retried. P6.5 assesses whether retry logic is safe and bounded; P6.4 assesses whether failure after exhausted retries is handled gracefully.
- [P6.2](P6.2-error-classification.md) — Resilience depends on classification. The system must classify infrastructure errors as non-critical to handle them differently from critical failures.

## Boundary Notes

- The distinction between critical and non-critical is context-dependent. A database may be critical for persisting run results but a cache may be non-critical. The assessment checks whether this distinction exists and is handled, not whether the specific classification is optimal.
- P6.4 does not require the system to tolerate failures of critical dependencies. If the LLM provider is down, the agent cannot execute — that is expected. P6.4 assesses whether non-critical failures are isolated from the critical path.
- A system that has no non-critical dependencies (everything it calls is essential) is N/A for P6.4, not a GAP. The assessment must determine whether the "everything is critical" claim is genuine or whether non-critical dependencies exist but are not recognized as such.
