## Intent

Transport-layer data shapes must be separate from internal domain models. API request/response types, serialization formats, and wire protocol structures must not be used as internal domain types. The mapping between transport shapes and domain models must be explicit.

## Why It Matters

When API response shapes are used directly as internal domain models, the system's internal architecture becomes coupled to its external interface. Changing an API field name requires changing internal code. Adding an internal field exposes it in the API. Deprecating an API field requires ensuring no internal code depends on it. Every API evolution becomes an internal refactoring exercise.

DTO isolation creates a buffer between external contracts and internal models. The API can evolve (adding fields, changing formats, supporting multiple versions) without touching internal logic. Internal models can evolve (adding computed fields, changing representations) without affecting the API. Each layer changes independently, with an explicit mapping layer mediating between them.

## Applicability

Applies to: Systems that expose or consume HTTP APIs, GraphQL endpoints, gRPC services, or any transport-layer interface with defined request/response shapes
Requires: At least one transport-layer interface with structured request/response types

## N/A Conditions

N/A when the system has no HTTP API layer, no external-facing transport interface, and all interactions occur through direct function calls within a single process. Also N/A for CLI-only tools with no network interface.

## Required Behaviors

1. API request types (DTOs, view models, request schemas) are defined separately from internal domain models
2. API response types are defined separately from internal domain models
3. An explicit mapping exists between DTOs and domain models (conversion functions, constructors, serializers)
4. Internal domain logic does not import or reference DTO types
5. API layer code does not operate directly on domain model internals beyond what is needed for mapping

## Acceptable Implementation Patterns

- Separate model directories for API models and domain models with explicit mapping functions
- Serializer layer that converts between domain objects and API representations
- Request/response wrapper types in the API layer that map to domain types via constructors or factory functions
- GraphQL resolver layer that maps between graph types and domain types
- Auto-mapping libraries configured to convert between DTO and domain types with explicit field mapping declarations

## Evidence to Seek

- **Runtime evidence:** API responses that contain a subset or transformation of internal domain data, not a direct serialization of domain objects
- **Trace evidence:** Follow data from an API endpoint handler through mapping to the domain layer, verifying the DTO and domain types are distinct
- **Implementation evidence:** Separate type definitions for API models and domain models, mapping functions or serializers, import analysis showing domain modules do not import API types
- **Declarative evidence:** API schema documentation (OpenAPI, GraphQL schema) that differs from internal model documentation (lowest weight)

## Trace Method

1. Find the API layer (HTTP handlers, GraphQL resolvers, gRPC service implementations)
2. Identify the types used in API request/response handling
3. Check whether these types are the same objects used in the domain/runtime layer or distinct types
4. If distinct, find the mapping code that converts between API types and domain types
5. Check domain module imports — do they reference API types?
6. Check whether adding an internal field to a domain model would automatically expose it in the API (if yes, isolation is absent)

## Verdict Conditions

**PASS when:** API types and domain types are distinct AND explicit mapping exists between them AND domain modules do not import API types AND the API layer can evolve independently of internal models.

**PARTIAL when:** Some API endpoints use separate DTOs while others return domain models directly, OR DTOs exist but the mapping is incomplete (some fields pass through without transformation), OR separate types exist but domain code imports DTO types for convenience.

**GAP when:** Domain models are used directly as API request/response types AND no mapping layer exists AND internal model changes directly affect the API surface.

**UNASSESSABLE when:** The system uses a framework that auto-serializes domain models to API responses with configurable field inclusion/exclusion, and it cannot be determined from static analysis whether this constitutes effective isolation or direct exposure.

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether the mapping between DTOs and domain models is complete and correct (e.g., whether all sensitive internal fields are excluded from API responses). However, verifying the structural separation and existence of mapping code is sufficient for static assessment.

## Boundary Notes

- P5.3 addresses transport-layer isolation specifically. Internal module-to-module interfaces within the domain layer are not DTOs and are not governed by P5.3.
- A system that only has an internal API (no external consumers) may still benefit from DTO isolation, but the severity is lower since the coupling is internal.
- When the domain model and the API model are genuinely identical (e.g., a simple CRUD application with no business logic), the absence of DTO isolation is a design choice, not necessarily a gap. However, in agentic systems with complex domain models, this situation is rare.
- Event payloads that leave the system boundary (published to external queues, webhooks) are transport-layer shapes and should be treated as DTOs for the purpose of P5.3.
