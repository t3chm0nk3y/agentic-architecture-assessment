## Intent

LLM outputs that feed downstream systems must be validated against a schema before use. The system must treat LLM responses as untrusted external input — probabilistic, potentially malformed, and not guaranteed to conform to the expected structure.

## Why It Matters

LLMs produce output based on statistical patterns, not contractual guarantees. A prompt that asks for JSON may receive JSON with extra fields, missing fields, incorrect types, or trailing text. A prompt that asks for a specific enum value may receive a paraphrase. A prompt that asks for a structured plan may receive a narrative paragraph. These deviations are not bugs in the LLM — they are inherent to the technology.

When LLM output flows directly into downstream systems without validation, failures are unpredictable and hard to diagnose. A missing field causes a null reference in step execution. An unexpected type causes a serialization error in event emission. A malformed tool call causes a tool failure that is attributed to the tool rather than to the LLM output. Validating LLM output at the point of ingestion converts these scattered failures into clear, located rejections with actionable error messages.

## Applicability

Applies to: All systems that use LLM output to drive downstream behavior (tool selection, step execution, data extraction, decision making)
Requires: At least one LLM call whose output feeds a downstream system component

## N/A Conditions

N/A when LLM output is only presented to humans (e.g., a chatbot that displays LLM responses directly) and never feeds downstream automated processing. If the LLM response is the final output and nothing parses or acts on its structure, validation is not required.

## Required Behaviors

1. LLM responses that drive downstream behavior are parsed and validated against a defined schema before use
2. Validation checks structural conformance (required fields present, correct types, valid enum values)
3. Validation failures produce structured error records that include what was expected and what was received
4. Validation failures are handled gracefully — the system retries, falls back, or fails with a clear error rather than propagating malformed data
5. The expected output schema is defined explicitly (as a code type, JSON schema, or equivalent), not implicitly assumed by downstream consumers

## Acceptable Implementation Patterns

- Structured output parsing with validation (e.g., parsing LLM JSON output into a validated typed model)
- Schema-validated extraction where the LLM response is validated against a JSON schema before downstream use
- Retry-on-validation-failure pattern where a failed validation triggers a retry with feedback to the LLM
- Response wrapper that parses, validates, and returns a typed result or a structured error
- Provider-level structured output mode (where the LLM provider guarantees schema conformance) combined with application-level validation as a defense-in-depth measure
- Grammar-constrained or guided decoding that restricts LLM output to valid schema instances, with post-generation validation as confirmation

## Evidence to Seek

- **Runtime evidence:** Validation error logs showing LLM outputs that were rejected for schema violations, demonstrating the validation pathway is active
- **Trace evidence:** Follow an LLM response from the provider adapter through parsing and validation to the point where validated data enters the runtime
- **Implementation evidence:** LLM response parsing code with schema validation, output schema definitions, error handling for validation failures, retry logic for malformed outputs
- **Declarative evidence:** Documentation describing expected LLM output formats (lowest weight — must be backed by implementation)

## Trace Method

1. Find all LLM invocation points in the system (provider calls, completion requests, chat calls)
2. For each invocation, trace the response from receipt to first downstream use
3. Check whether the response is parsed into a typed/validated structure between receipt and use
4. Check whether the parsing/validation uses an explicit schema definition
5. Check what happens when validation fails — is there error handling, retry logic, or does the failure propagate unhandled?
6. Verify that the validated output (not the raw response) is what downstream systems consume
7. Check for edge case handling: what happens when the LLM returns empty output, partial JSON, or unexpected format?

## Verdict Conditions

**PASS when:** All LLM outputs that drive downstream behavior are validated against explicit schemas AND validation failures are handled gracefully (retry, fallback, or clear error) AND downstream systems consume validated output, not raw LLM responses.

**PARTIAL when:** Validation exists for some LLM output pathways but not others, OR validation checks structure but not semantic constraints (e.g., checks for JSON validity but not field types), OR validation exists but failures are not handled gracefully (e.g., raw exception on validation failure).

**GAP when:** LLM output is used directly without schema validation AND downstream systems parse raw LLM responses inline AND no explicit output schema is defined.

**UNASSESSABLE when:** The system uses an LLM provider's structured output feature that may guarantee schema conformance at the provider level, but application-level validation cannot be confirmed from the codebase.

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether the validation schemas are comprehensive enough to catch all real-world LLM output variations (LLMs can produce unexpected output in ways that are hard to anticipate). However, the presence of explicit schemas, validation code, and failure handling is sufficient for static assessment.

## Boundary Notes

- P5.5 applies to LLM outputs that drive automated downstream behavior. LLM outputs that are displayed to users without automated processing are not in scope.
- Provider-level structured output (e.g., JSON mode, function calling with schema enforcement) reduces the risk but does not eliminate the need for application-level validation. Provider guarantees may change, and defense-in-depth is a sound practice.
- Prompt engineering that instructs the LLM to produce valid output is not a substitute for validation. Instructions reduce the frequency of malformed output but do not eliminate it.
- The severity of validation failure handling depends on the downstream impact. A failed extraction for a logging field may be acceptable to skip; a failed extraction for a tool invocation argument must block execution. P5.5 requires that failures are handled, but the appropriate handling is context-dependent.
