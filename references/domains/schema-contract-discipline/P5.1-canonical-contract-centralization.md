## Intent

System-wide data entities must be defined once in a single authoritative location. No duplicate definitions across modules. When multiple modules need the same data shape, they import it from the canonical source rather than redefining it locally.

## Why It Matters

Duplicate entity definitions are a latent source of bugs that compound over time. When the same concept (a run, a step, a tool result, a user, a workflow) is defined in two places, the definitions inevitably diverge. One gets a new field, the other doesn't. One changes a field type, the other keeps the old type. Data flowing between modules silently loses fields, gains unexpected nulls, or triggers type errors far from the root cause.

Centralizing definitions means a single change propagates everywhere. It makes the system's data model auditable — you can enumerate all entities by reading one location. It eliminates an entire class of bugs where "the data was right at the source but wrong at the destination because the destination expected a different shape."

## Applicability

Applies to: All systems with shared data entities used across multiple modules
Requires: At least two modules that operate on the same data entity

## N/A Conditions

N/A when the system is a single module with no shared data entities — all definitions are inherently centralized. This is rare in any system of meaningful complexity.

## Required Behaviors

1. Core data entities (runs, steps, tool results, events, configurations, etc.) are defined in an identifiable, shared location
2. Modules that use these entities import the definitions from the shared location rather than redefining them
3. No module contains a local redefinition of a core entity that duplicates or partially duplicates a centrally-defined entity
4. The canonical definitions are the types used in inter-module interfaces (function signatures, API contracts, event payloads)
5. When an entity evolves, the change is made in the canonical location and propagated, not patched locally in consuming modules

## Acceptable Implementation Patterns

- Shared models or schemas package/module that all other modules depend on for entity definitions
- Domain model layer in a layered architecture where entities are defined once and imported by services, adapters, and API layers
- Protobuf, JSON Schema, or equivalent schema definition files that are the single source from which code is generated
- Type aliases or re-exports from a central module for entities that originate in a dependency
- Monorepo shared types package that multiple services import

## Evidence to Seek

- **Runtime evidence:** Serialized data at module boundaries matching the canonical schema — no extra or missing fields from local redefinitions
- **Trace evidence:** Follow a core entity from its canonical definition through imports in consuming modules, verifying they reference the same definition
- **Implementation evidence:** A models or schemas directory/module, import statements in consuming modules pointing to the canonical location, absence of duplicate entity definitions elsewhere
- **Declarative evidence:** Architecture documentation describing the canonical model location (lowest weight)

## Trace Method

1. Identify core data entities in the system (the nouns: runs, steps, tools, events, results, etc.)
2. For each entity, search for its definition across the entire codebase
3. Check whether each entity is defined in exactly one location or in multiple locations
4. If defined in one location, verify that consuming modules import from that location
5. If defined in multiple locations, check whether the definitions are identical, overlapping, or divergent
6. Check inter-module interfaces (function parameters, return types, event payloads) — do they reference the canonical types or local types?

## Verdict Conditions

**PASS when:** Core data entities are defined in an identifiable shared location AND consuming modules import from the canonical source AND no duplicate definitions exist across modules.

**PARTIAL when:** A canonical location exists for most entities, but some entities are duplicated across modules, OR the canonical definitions exist but some modules redefine entities locally (even if currently identical).

**GAP when:** No canonical entity location is identifiable AND data entities are defined independently in each module that needs them AND the same conceptual entity has divergent definitions across the codebase.

**UNASSESSABLE when:** The system uses dynamic typing or dictionary-based data passing throughout, making it impossible to identify entity definitions statically.

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether the canonical definitions are actually the types used at runtime (e.g., runtime code could construct entities from dictionaries that happen to match the canonical shape). However, verifying centralized definitions and consistent imports is sufficient for static assessment.

## Boundary Notes

- Intentionally separate types for different purposes (e.g., a domain model, a DTO, and a database model for the same entity) are not violations of P5.1. P5.1 governs unintentional duplication where the same conceptual type is defined redundantly. Intentional separation with explicit mapping is a design pattern (see P5.3).
- Generated code (e.g., from protobuf or OpenAPI specs) that produces types in multiple output directories from a single source definition satisfies P5.1 — the source definition is canonical even if the generated output appears in multiple places.
- Test-specific data types (test fixtures, mock types) that mirror production types are not P5.1 violations, though they become a maintenance burden if they diverge.
