## Intent

Workflow templates, skill templates, step templates must be clearly separated from runtime records. A definition describes what should happen; a runtime record describes what did happen. These are distinct data types with distinct lifecycles and must not be conflated into a single structure.

## Why It Matters

Definitions and runtime records serve fundamentally different purposes. A workflow definition is a reusable blueprint — it describes the steps, the ordering, the parameters. A workflow run is a specific execution of that blueprint — it has a start time, a status, step results, and an outcome. When these are conflated into a single type, the system cannot distinguish between modifying a definition (which affects future executions) and modifying a runtime record (which tracks a specific past execution).

This conflation creates practical problems: Can you re-run a workflow from its definition if the definition also contains runtime state from the last run? Can you enumerate all available workflow definitions without seeing every past execution? Can you version a definition independently of its execution history? Separation makes all of these operations straightforward.

## Applicability

Applies to: Systems that have reusable definitions, templates, or blueprints that are instantiated into runtime executions
Requires: At least one entity that exists as both a template/definition and a runtime instance

## N/A Conditions

N/A when the system is a fully autonomous agent with no predefined workflows, skills, or step templates. If the agent constructs its entire execution plan dynamically with no reusable definitions, the definition/runtime distinction does not apply. Also N/A for systems that are purely reactive and never reuse execution patterns.

## Required Behaviors

1. Definition types (templates, blueprints, configurations) are distinct data types from runtime types (runs, executions, records)
2. Definitions do not contain runtime state fields (status, start time, results, errors)
3. Runtime records reference their source definition but do not contain the full definition inline (or if they do, the definition snapshot is a separate embedded field, not mixed with runtime fields)
4. Definitions have their own lifecycle (creation, versioning, deletion) independent of runtime record lifecycle
5. The system's data model makes the definition/runtime boundary explicit and unambiguous

## Acceptable Implementation Patterns

- Separate type hierarchies for definitions and runtime records (e.g., WorkflowDefinition and WorkflowRun as distinct classes)
- Definition tables and runtime tables in the data layer, with runtime records referencing definitions by ID
- Template pattern where a definition is instantiated into a runtime record via a factory or constructor
- Configuration-as-code where definitions are in files and runtime records are in a database or event store
- Immutable definitions with versioning, and runtime records pinned to a specific definition version

## Evidence to Seek

- **Runtime evidence:** Database or storage inspection showing separate definition and runtime record collections with different schemas
- **Trace evidence:** Follow a definition from its storage through instantiation into a runtime record, verifying the two are distinct objects with distinct types
- **Implementation evidence:** Separate type definitions for definition entities and runtime entities, factory or instantiation code that creates a runtime record from a definition, no runtime state fields on definition types
- **Declarative evidence:** Data model documentation distinguishing definitions from runtime records (lowest weight)

## Trace Method

1. Identify the system's core reusable entities (workflow definitions, skill definitions, step templates, agent configurations)
2. Find the type definitions for these entities
3. Check whether the type includes both definition fields (name, steps, configuration) and runtime fields (status, start_time, results)
4. If combined, the types are conflated — P5.2 gap
5. If separate, verify that runtime records reference the definition (by ID, version, or embedded snapshot) rather than duplicating definition fields
6. Check the data layer — are definitions and runtime records stored in separate collections/tables?
7. Verify that definitions can exist without runtime records (i.e., a definition can be created before it is ever executed)

## Verdict Conditions

**PASS when:** Definition types and runtime types are distinct AND definitions do not contain runtime state fields AND runtime records reference their source definition AND the two have independent lifecycles.

**PARTIAL when:** Definitions and runtime records are partially separated — for example, separate types exist but some runtime fields have leaked into the definition type, OR runtime records embed the full definition rather than referencing it.

**GAP when:** A single type serves as both definition and runtime record AND there is no structural separation between template fields and execution state fields AND the same object is used for both defining a workflow and tracking its execution.

**UNASSESSABLE when:** The system uses dynamic or untyped data structures (dictionaries, JSON blobs) that make it impossible to determine whether definition and runtime concerns are separated.

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether the separation holds at runtime when data flows through the system (e.g., whether runtime code mutates definition objects). However, verifying distinct type definitions and separate storage is sufficient for static assessment.

## Boundary Notes

- P5.2 does not prescribe how definitions are stored (files, database, configuration). It only requires that definitions are structurally distinct from runtime records regardless of storage mechanism.
- A runtime record that includes a snapshot of the definition it was instantiated from (for auditability) is acceptable as long as the snapshot is a clearly demarcated embedded field, not mixed with runtime fields.
- Systems that dynamically generate execution plans (e.g., LLM-generated step sequences) may not have persistent definitions, but if the generated plan is stored as a template for replay or review, it becomes a definition and P5.2 applies.
- Versioning of definitions is a best practice but not strictly required by P5.2. The principle requires separation, not version management.
