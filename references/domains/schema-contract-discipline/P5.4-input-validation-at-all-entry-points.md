## Intent

All external inputs — API requests, tool inputs, configuration files, definition files, environment variables, and any other data entering the system from outside its trust boundary — must be validated against a schema before use. Invalid data must be rejected with a clear, actionable error before it reaches runtime logic.

## Why It Matters

Unvalidated input is the root cause of a vast category of runtime failures. A missing required field causes a null reference error three layers deep. A string where a number was expected causes a type error in a calculation. A malformed configuration file causes the system to start with incorrect behavior that manifests hours later. In every case, the failure is far from the source, making diagnosis expensive.

Validation at entry points converts these obscure runtime failures into clear, immediate rejections. When every external input is validated against a schema at the boundary, malformed data never enters the runtime. Errors are specific ("field X is required, got null"), located ("in API request to /runs"), and actionable ("provide a non-null string for field X").

## Applicability

Applies to: All systems that accept input from external sources
Requires: At least one external input pathway (API endpoint, configuration file, tool input, environment variable)

## N/A Conditions

None — every system has external inputs, even if only configuration or environment variables. Input validation applies universally.

## Required Behaviors

1. API request bodies are validated against a schema before the request handler processes them
2. Tool inputs (arguments passed to tool functions) are validated against the tool's declared input schema before execution
3. Configuration files are validated against a schema at load time, before configuration values are used
4. Definition files (workflow definitions, skill definitions, step templates) are validated against a schema at load time
5. Environment variables used for configuration are validated for presence, type, and format at startup
6. Validation errors produce structured, actionable error messages that identify the specific field(s) and constraint(s) that failed

## Acceptable Implementation Patterns

- Request validation middleware that validates incoming API requests against a schema before passing to handlers
- Pydantic, Zod, JSON Schema, or equivalent validation libraries applied to all external input types
- Configuration validation at application startup using a typed configuration schema
- Tool input validation within the tool execution framework, applied before tool function invocation
- Schema-based parsing where external data is parsed into validated typed objects (parse, don't validate)
- Contract-first API design where the OpenAPI/GraphQL schema drives both validation and documentation

## Evidence to Seek

- **Runtime evidence:** Validation error responses from the API showing structured rejection of malformed input; startup failures with clear messages when configuration is invalid
- **Trace evidence:** Follow an external input from the entry point to the first point of use, verifying that validation occurs between receipt and use
- **Implementation evidence:** Validation schemas or models for API requests, tool inputs, configuration, and definitions; validation middleware or validation calls at entry points; structured error responses for validation failures
- **Declarative evidence:** API schema definitions (OpenAPI, GraphQL) with validation constraints; configuration schema documentation (lowest weight)

## Trace Method

1. Identify all external input pathways (API endpoints, configuration loaders, definition file loaders, tool input paths, environment variable reads)
2. For each API endpoint, check whether the request body is validated against a schema before the handler processes it
3. For each tool, check whether tool inputs are validated against the declared input schema before tool execution
4. For configuration loading, check whether configuration values are validated against a schema at load time
5. For definition file loading, check whether definition content is validated against a schema before use
6. For environment variables, check whether they are validated for presence and type at startup
7. For each pathway, verify that validation failure produces a structured error that does not proceed to runtime processing

## Verdict Conditions

**PASS when:** All identified external input pathways have schema-based validation AND validation occurs before data reaches runtime logic AND validation errors produce structured, actionable messages.

**PARTIAL when:** Validation exists for some input pathways but not others (e.g., API inputs are validated but configuration files are not), OR validation exists but is incomplete (e.g., presence is checked but type and format are not).

**GAP when:** No systematic input validation exists AND external data flows directly into runtime logic without schema checks AND malformed input causes obscure downstream errors rather than clear boundary rejections.

**UNASSESSABLE when:** A framework or platform layer handles validation implicitly (e.g., auto-generated API validation from a schema), but the validation behavior cannot be confirmed from the codebase.

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether validation is complete — i.e., whether all fields and constraints are covered (a schema could validate presence but miss format constraints). However, the presence of systematic validation at entry points is sufficient for static assessment.

## Boundary Notes

- P5.4 assesses whether validation exists at entry points, not whether the validation schemas are correct or complete (schema correctness is a functional testing concern, not an architectural principle)
- Validation depth is a judgment call. Validating that a field is a string is minimal; validating that a field matches a specific pattern is thorough. P5.4 requires schema-based validation; the depth of the schema is an implementation detail.
- Internal function-to-function calls within the runtime do not require P5.4 validation. The principle applies at trust boundaries — where data enters from outside the system's control.
- Runtime type checking (e.g., Python type hints without runtime enforcement, TypeScript types without runtime validation) does not satisfy P5.4. Validation must be enforced at runtime, not just at compile time or in type annotations.
