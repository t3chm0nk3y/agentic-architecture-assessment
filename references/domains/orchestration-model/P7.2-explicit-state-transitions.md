## Intent

All run and step state transitions must be explicit, recorded, and non-skippable. The system must not silently move between states or allow states to be skipped.

## Why It Matters

State transitions are the backbone of execution auditability. When a run transitions from "pending" to "running" to "completed" (or "failed"), each transition represents a meaningful event in the execution lifecycle. If transitions are implicit (state changes as a side effect of other operations), unrecorded (no log or event marks the transition), or skippable (a run can jump from "pending" to "completed" without passing through "running"), the execution history becomes unreliable. Operators cannot determine what state the run was in when a failure occurred, monitoring systems cannot track progress, and post-hoc audits cannot reconstruct the execution sequence.

## Applicability

Applies to: Systems with multi-step or stateful execution
Requires: A state model for runs, steps, or execution phases

## N/A Conditions

N/A when the system is a stateless single-turn agent — a system that receives a request, performs a single operation, and returns a response without any intermediate states or steps. If there are no states to transition between, this principle does not apply.

## Required Behaviors

1. Run and step states are explicitly defined (enum, constants, or equivalent) rather than inferred from other fields
2. State transitions are performed through a dedicated mechanism (transition function, state machine, status update method) rather than by directly setting a status field
3. State transitions are recorded — each transition produces a log entry, event, or persistent record
4. State transitions are non-skippable — the system enforces that transitions follow valid paths (e.g., a run cannot go from "pending" to "completed" without passing through "running")
5. Invalid state transitions are rejected or flagged rather than silently accepted

## Acceptable Implementation Patterns

- State machine with defined states and valid transitions, enforcing transition rules at runtime
- Status update method or function that validates the transition, records it, and emits an event before changing the state
- State transition table or configuration that defines valid (from_state, to_state) pairs
- Event-sourced state where the current state is derived from a sequence of recorded transition events
- ORM or database-level constraints that enforce valid state values and transition sequences

## Evidence to Seek

- **Runtime evidence:** Execution logs or events showing state transitions with timestamps, from_state, to_state, and reason
- **Trace evidence:** Follow a run from creation to completion, checking that each state transition is explicit, recorded, and follows a valid path
- **Implementation evidence:** State enum definitions, transition functions/methods, transition validation logic, event emission at transition points
- **Declarative evidence:** State machine diagrams, state transition documentation

## Trace Method

1. Find the state model: look for run status enums, step status constants, or state field definitions
2. Find the transition mechanism: look for functions/methods that change state (update_status, transition, set_state)
3. Check whether transitions are validated — does the mechanism check that the transition is valid before applying it?
4. Check whether transitions are recorded — does the mechanism emit a log entry, event, or persistent record at transition time?
5. Search for direct state field mutations that bypass the transition mechanism (e.g., `run.status = "completed"` instead of `run.transition_to("completed")`)
6. Check for state skipping: can a run go from creation to completion without intermediate states? Look for code paths that set terminal states directly
7. Check error state transitions: when a failure occurs, does the system transition to an error/failed state explicitly, or does the state simply stop updating?

## Verdict Conditions

**PASS when:** States are explicitly defined AND transitions go through a dedicated mechanism that validates and records them AND no code path bypasses the transition mechanism to set states directly AND transitions follow valid paths.

**PARTIAL when:** States are defined and a transition mechanism exists but is not consistently used — some transitions go through the mechanism while others set state directly, OR transitions are recorded but not validated (invalid transitions are accepted), OR the mechanism exists for run-level states but not step-level states.

**GAP when:** No explicit state transition mechanism exists — states are set by directly mutating status fields throughout the codebase without validation or recording, OR run/step states are not defined at all (the system has no state model despite having multi-step execution).

**UNASSESSABLE when:** A state model and transition mechanism exist but the enforcement cannot be verified from static analysis (e.g., state transitions are enforced by database triggers or framework hooks not visible in the application code).

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether transition validation actually prevents all invalid transitions at runtime (there may be race conditions or edge cases). However, the existence of explicit states, a transition mechanism with validation, and recording at transition points is sufficient for static assessment.

## Boundary Notes

- P7.2 assesses the transition mechanism, not the quality or granularity of the state model. A simple three-state model (pending, running, completed/failed) with proper transitions is PASS; a granular ten-state model with direct field mutations is GAP.
- Direct state field access for reading is not a concern — P7.2 is about mutations/transitions, not reads.
- The requirement for "non-skippable" transitions means that the system enforces valid transition paths. It does not mean every system must have the same state model. If a system's state model has only two states (running, done), a direct transition from running to done is valid — it is not "skipping" a state.
