## Intent

The system must record enough state to audit any run after the fact. Given a completed run, it must be possible to reconstruct what happened: what steps executed, in what order, what the inputs and outputs were, what decisions were made, and what the final outcome was.

## Why It Matters

Agentic systems make decisions, invoke tools with real-world effects, and produce outputs that may be consumed by humans or downstream systems. When something goes wrong — a run produces an incorrect result, a tool is invoked with harmful parameters, or a run fails unexpectedly — operators need to reconstruct the execution after the fact. If run state is ephemeral (exists only in memory during execution), the system becomes a black box once execution completes. Post-hoc auditing, incident investigation, compliance reporting, and system improvement all depend on persisted run state.

## Applicability

Applies to: All systems where runs have consequences that outlast the execution itself
Requires: Any form of persistent state storage or logging infrastructure

## N/A Conditions

N/A when the system is explicitly ephemeral with no persistence requirement — a system that is architecturally designed to retain no state and whose callers accept that runs cannot be audited. This must be an explicit architectural decision, not an oversight. Note: this exemption is rare in production systems and does not apply merely because persistence has not been implemented yet.

## Required Behaviors

1. The system persists a record for each run that includes at minimum: run identifier, start time, end time, final outcome (success, failure, truncated), and the initiating input or trigger
2. Step-level records are persisted: which steps executed, in what order, with what inputs and outputs
3. Tool invocation records are persisted: which tools were called, with what parameters, and what results were returned
4. Decision points are recorded: when the system made a choice (which tool to call, whether to continue, which branch to take), the decision and its basis are persisted
5. The persisted state is sufficient to reconstruct the execution sequence without access to runtime memory — an auditor can understand what happened from the persisted records alone
6. Run records are accessible after execution completes — they are not garbage-collected, overwritten, or deleted immediately

## Acceptable Implementation Patterns

- Run and step records in a database with structured fields for inputs, outputs, tool calls, and decisions
- Event-sourced run history where the complete execution can be reconstructed from a sequence of events
- Structured log output with run-scoped entries that capture step execution, tool calls, and decisions (acceptable when logs are reliably persisted and searchable)
- Run artifact storage (JSON files, S3 objects) containing the complete execution trace
- Append-only execution log with structured entries per step and tool call
- Combination of database records (summary) and log/event stream (detail)

## Evidence to Seek

- **Runtime evidence:** A completed run whose execution can be fully reconstructed from persisted records — step sequence, tool calls, decisions, and outcome
- **Trace evidence:** Follow a run from start to completion and check what is persisted at each stage — are step records created? Are tool call records created? Are decisions recorded?
- **Implementation evidence:** Run record creation code, step record creation code, tool call logging code, decision recording code, persistence layer (database writes, event emission, log output)
- **Declarative evidence:** Data model documentation showing run/step/tool call record schemas

## Trace Method

1. Find the run lifecycle code: where is a run created, updated, and finalized?
2. Check what is persisted at run creation: run ID, input, start time, configuration
3. Check what is persisted during execution: step records, tool call records, intermediate state
4. Check what is persisted at run completion: final outcome, end time, output/result
5. Assess completeness: given only the persisted records (no runtime memory, no external logs), can an auditor reconstruct the execution sequence?
6. Check persistence durability: are records written to a durable store (database, file system, event log) or only held in memory?
7. Check record retention: are records accessible after the run completes, or are they cleaned up?
8. Look for gaps: are there execution phases, tool calls, or decision points that do not produce persistent records?

## Verdict Conditions

**PASS when:** Run records with identifier, timing, outcome, and input are persisted AND step-level execution records are persisted AND tool call records (parameters and results) are persisted AND the persisted state is sufficient to reconstruct the execution sequence after the fact.

**PARTIAL when:** Some run state is persisted but not enough for full reconstruction: run-level records exist but step-level detail is missing, OR tool call parameters are recorded but results are not, OR decisions are not recorded (only actions), OR persistence is inconsistent (some runs fully recorded, others partially).

**GAP when:** No run state is persisted — execution state exists only in memory and is lost when the run completes, OR only a final status is persisted without any step-level, tool-level, or decision-level detail.

**UNASSESSABLE when:** Persistence code exists but the durability and completeness of persisted records cannot be determined from static analysis (e.g., records are written to a logging framework whose persistence configuration is external to the codebase).

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether persisted records actually contain meaningful data at runtime (fields could be empty or placeholder values). However, the persistence code, record structure, and write points throughout the execution lifecycle are sufficient for static assessment.

## Boundary Notes

- P7.5 assesses whether enough state is persisted for post-hoc auditing, not whether the system has a formal audit framework or compliance certification. A system with comprehensive structured logs that capture the full execution is PASS even without a dedicated audit feature.
- The level of detail required depends on the system's risk profile. A system that sends emails via tool calls requires more detailed tool call recording than a system that only reads data. The assessment should consider consequences when evaluating completeness.
- Structured logs are an acceptable persistence mechanism as long as they are reliably persisted, searchable by run ID, and contain structured fields (not free-form text). Unstructured console output that happens to contain execution information is PARTIAL at best.
- P7.5 does not require real-time audit capability (that is more P3.x observability). P7.5 is specifically about post-hoc reconstruction — can you audit the run after it has completed?
