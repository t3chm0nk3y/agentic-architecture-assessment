## Intent

Structured workflows must be deterministic — the steps that will execute are known before execution begins. Given the same workflow definition and inputs, the same steps execute in the same order.

## Why It Matters

When a system uses structured workflows (as opposed to fully autonomous agent loops), operators expect predictability: they define a sequence of steps, and those steps execute in that sequence. If workflow execution is non-deterministic — steps execute in unpredictable orders, steps are silently skipped, or the execution path varies without clear rules — operators cannot reason about what a workflow will do, testing becomes unreliable (different runs produce different step sequences), and debugging requires reconstructing which path was taken rather than knowing it in advance. Deterministic workflow structure is what distinguishes a workflow from an ad-hoc script.

## Applicability

Applies to: Systems with defined, structured workflows (pipelines, step sequences, DAGs)
Requires: A workflow concept where steps are defined before execution

## N/A Conditions

N/A when the system is fully autonomous with no workflow concept — an agent that dynamically decides every action through LLM reasoning without predefined steps. In such systems, P7.3 (Bounded Autonomous Execution) governs the execution model instead. Note: a system can have both workflows AND autonomous steps within a workflow; in that case, P7.4 applies to the workflow structure while P7.3 applies to the autonomous steps.

## Required Behaviors

1. Workflow steps are defined before execution begins — the set of steps that will execute is known from the workflow definition
2. Given the same workflow definition and input conditions, the same steps execute in the same order
3. Conditional steps (if they exist) have explicit conditions that are evaluable before or during execution — the branching logic is part of the workflow definition, not emergent behavior
4. Steps are not silently skipped or reordered at runtime without an explicit, defined mechanism (e.g., a skip condition in the workflow definition)
5. The workflow execution engine faithfully executes the defined workflow — it does not add, remove, or reorder steps beyond what the definition specifies

## Acceptable Implementation Patterns

- Sequential pipeline where steps execute in definition order, one after another
- Directed acyclic graph (DAG) with explicit dependency edges, where steps execute when dependencies are satisfied
- Workflow definition language or configuration (YAML, JSON, DSL) that declares steps and their ordering
- Step-runner that iterates over a list of step definitions and executes each in sequence
- Conditional branching with explicit branch conditions defined in the workflow (if condition X, execute step A; otherwise, execute step B)
- Parallel execution of independent steps within a defined workflow, with a deterministic merge point

## Evidence to Seek

- **Runtime evidence:** Two executions of the same workflow with the same inputs producing the same step sequence (or deterministically different sequences based on explicit conditions)
- **Trace evidence:** Follow a workflow definition from declaration through the execution engine to runtime — verify that the engine executes exactly the defined steps in the defined order
- **Implementation evidence:** Workflow definition structures (step lists, DAG edges, pipeline configurations), execution engine code that iterates over definitions, conditional branching logic
- **Declarative evidence:** Workflow documentation showing step sequences and conditions

## Trace Method

1. Find the workflow definition mechanism — how are workflows defined? (Code, configuration, DSL)
2. Identify a specific workflow definition — what steps does it declare? In what order?
3. Find the workflow execution engine — the code that reads a workflow definition and executes it
4. Trace the engine's execution of the defined workflow — does it iterate over the defined steps? Does it respect the defined order?
5. Check for runtime modifications: does the engine add, remove, or reorder steps based on runtime conditions not in the workflow definition?
6. Check for conditional steps: if branches exist, are the conditions explicit and part of the workflow definition?
7. Check for non-deterministic elements: random ordering, race-condition-dependent ordering, or LLM-decided ordering within the workflow (as opposed to within an autonomous step)

## Verdict Conditions

**PASS when:** Workflow steps are defined before execution AND the execution engine faithfully executes them in the defined order AND conditional branching (if present) has explicit conditions in the workflow definition AND no silent skipping or reordering occurs.

**PARTIAL when:** Workflow structure is mostly deterministic but has limited non-deterministic elements: some steps can be reordered or skipped based on runtime conditions not in the workflow definition, OR parallel step execution exists without deterministic merge behavior, OR the workflow engine mostly follows definitions but has framework-injected steps.

**GAP when:** Workflow execution is non-deterministic — steps execute in unpredictable order, steps are silently skipped or added at runtime, OR the workflow definition does not determine the execution path (the engine makes ad-hoc decisions about what to execute).

**UNASSESSABLE when:** A workflow definition exists but the execution engine's behavior cannot be determined from static analysis (e.g., the engine is a framework whose execution semantics are not visible in the codebase).

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether the execution engine introduces non-determinism under specific runtime conditions (e.g., race conditions in parallel execution, timing-dependent step ordering). However, the workflow definition structure, the engine's execution logic, and the absence of runtime modification code are sufficient for static assessment.

## Boundary Notes

- P7.4 assesses determinism of the workflow structure, not determinism of step outputs. A workflow that always executes steps A, B, C in order is deterministic even if step B produces different outputs on different runs (that is the nature of LLM-based steps).
- Conditional branching is compatible with determinism — a workflow that executes step A, then branches to B or C based on an explicit condition, is deterministic. The key requirement is that the branching logic is defined in the workflow, not emergent.
- Fully autonomous agents where every action is LLM-decided are N/A for P7.4 — they do not have a workflow to be deterministic. This is not a gap; it is a different orchestration model governed by P7.3.
- A system may have multiple orchestration modes (scripted workflows and autonomous execution). P7.4 applies to the workflow mode; P7.3 applies to the autonomous mode. Both must be assessed.
