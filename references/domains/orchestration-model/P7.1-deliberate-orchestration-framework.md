## Intent

The orchestration approach must be a deliberate architectural choice. The system must have a coherent, identifiable control structure for agent execution rather than ad-hoc, scattered control flow.

## Why It Matters

Agentic systems require a control structure that determines how execution proceeds: what steps run, in what order, under what conditions, and with what constraints. When this control structure is a deliberate design choice — whether a state machine, a pipeline, an agent loop, or a DAG — the system's behavior can be understood, modified, and debugged. When orchestration is ad-hoc (scattered if/else chains, implicit ordering through function call sequences, control flow embedded in business logic), the system's behavior becomes emergent rather than designed. Emergent orchestration is difficult to reason about, test, and audit.

## Applicability

Applies to: All system types
Requires: Any form of multi-step or iterative execution

## N/A Conditions

N/A when the system is genuinely single-step with no iteration, no multi-step workflow, and no agent loop — a simple request-response transformation with no orchestration to assess. This is extremely rare in systems that warrant an agentic audit.

## Required Behaviors

1. The system has an identifiable orchestration component or pattern — a place in the codebase where execution flow is controlled
2. The orchestration approach is coherent — it follows a recognizable pattern (state machine, pipeline, agent loop, DAG, event-driven) rather than scattered, inconsistent control flow
3. The orchestration component has clear boundaries — it is distinguishable from business logic, tool implementations, and infrastructure code
4. The orchestration approach handles the system's core execution paths — it is not limited to a happy-path while edge cases and errors fall through to ad-hoc handling
5. The orchestration choice is appropriate for the system's complexity — overly simple orchestration for complex behavior is a potential concern, though not necessarily a gap

## Acceptable Implementation Patterns

- State machine with defined states and transitions governing run lifecycle
- Pipeline or step-runner that executes a defined sequence of operations
- Agent loop with clear structure: receive input, reason, select tool, execute, evaluate, decide to continue or stop
- Directed acyclic graph (DAG) execution engine with dependency resolution
- Event-driven orchestration with defined event handlers for lifecycle transitions
- Framework-provided orchestration (workflow engine, agent framework) with configuration-driven behavior

## Evidence to Seek

- **Runtime evidence:** Execution logs or traces showing a coherent flow pattern (consistent step ordering, state transitions, iteration patterns)
- **Trace evidence:** Follow an execution from entry point through to completion — is the control flow centralized and coherent, or scattered across modules?
- **Implementation evidence:** Orchestration module or class, state machine definitions, pipeline configurations, agent loop implementations
- **Declarative evidence:** Architecture documentation describing the orchestration approach

## Trace Method

1. Find the execution entry point (API handler, CLI entry, main function)
2. Trace the control flow from entry through to completion — identify where execution decisions are made (what runs next, when to stop, how to handle branches)
3. Assess whether the control flow is centralized in an identifiable orchestration component or scattered across multiple modules
4. Identify the orchestration pattern: is it a state machine, pipeline, agent loop, DAG, event-driven, or something else?
5. Check whether the pattern is coherent — does it handle all execution paths (happy path, error paths, edge cases), or only the happy path?
6. Check whether the orchestration component has clear boundaries — can you identify where orchestration ends and business logic begins?

## Verdict Conditions

**PASS when:** An identifiable orchestration component or pattern exists AND it is coherent (follows a recognizable pattern) AND it handles the system's core execution paths including error cases AND it has clear boundaries separating orchestration from business logic.

**PARTIAL when:** An orchestration pattern is identifiable but incomplete — it handles the happy path but errors or edge cases fall through to ad-hoc handling, OR the pattern exists but its boundaries with business logic are blurred.

**GAP when:** No coherent orchestration pattern is identifiable — execution flow is determined by scattered, ad-hoc control flow across multiple modules without a central control structure, AND this ad-hoc approach lacks clear bounded behavior.

**UNASSESSABLE when:** The system uses a framework-provided orchestration mechanism whose behavior cannot be assessed from the application code alone (e.g., orchestration is configured externally or delegated to a runtime not in the codebase).

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether the orchestration pattern actually produces coherent behavior at runtime under all conditions. However, the structural assessment of the orchestration component — its existence, coherence, and coverage — is sufficient for static assessment.

## Related Principles

- [P7.2](P7.2-explicit-state-transitions.md) — State transitions (P7.2) are a mechanism within the orchestration framework (P7.1). A deliberate orchestration framework provides the structure within which state transitions operate.
- [P7.3](P7.3-bounded-autonomous-execution.md) — Execution bounds (P7.3) are enforced by the orchestration framework. Ad-hoc orchestration makes it difficult to enforce bounds consistently.
- [P7.4](P7.4-deterministic-workflow-structure.md) — Workflow determinism (P7.4) is a property of the orchestration approach. Some orchestration patterns (pipelines, DAGs) are inherently deterministic; others (agent loops) require explicit bounds.

## Boundary Notes

- P7.1 assesses the existence and coherence of the orchestration approach, not its quality or sophistication. A simple but coherent agent loop is PASS; a complex but scattered control flow is GAP.
- "Ad-hoc orchestration without clear bounded behavior" is the gap condition. If ad-hoc orchestration demonstrably has bounded, predictable behavior despite lacking a formal framework, it may be a PARTIAL rather than a GAP. The assessment must consider whether the lack of formal orchestration actually causes problems.
- The choice of orchestration pattern (state machine vs. pipeline vs. agent loop) is not assessed — any deliberate, coherent pattern is acceptable. P7.1 does not prescribe a specific approach.
