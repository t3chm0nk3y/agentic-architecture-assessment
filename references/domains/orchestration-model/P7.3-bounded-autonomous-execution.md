## Intent

If autonomous agent execution exists, there must be explicit guardrails: maximum iterations, maximum tool calls, maximum token spend. The system must not be capable of executing indefinitely without constraint.

## Why It Matters

Autonomous agent execution — where an LLM iteratively reasons, selects tools, executes them, and decides whether to continue — is the defining capability and the primary risk of agentic systems. Without explicit bounds, an agent can loop indefinitely: consuming unlimited API tokens, making unlimited tool calls (potentially with real-world side effects), and occupying system resources without termination. Unbounded autonomous execution is not a theoretical risk — it is the default behavior of any agent loop that lacks termination constraints beyond the LLM deciding to stop. LLMs do not reliably self-terminate, and relying on LLM judgment for termination is insufficient.

## Applicability

Applies to: All systems with autonomous agent execution (LLM-driven iterative loops)
Requires: An execution path where the agent iterates autonomously — deciding whether to continue based on LLM output or tool results

## N/A Conditions

None — this principle applies to all agentic systems with autonomous execution. If the system has any form of iterative LLM-driven execution, explicit bounds are required. Systems with only fixed-step pipelines (no autonomous iteration) would assess P7.4 instead, but even pipelines with an LLM-driven "loop until done" phase require P7.3 bounds.

## Required Behaviors

1. Maximum iteration count is defined and enforced — the agent loop cannot execute more than a configured number of iterations
2. Maximum tool call count is defined and enforced — the total number of tool invocations within a single run is bounded
3. Maximum token spend is defined and enforced — the total token consumption (input + output) within a single run is bounded
4. When any bound is reached, the run terminates with a clear truncation indicator — not a silent stop, not a crash, and not continued execution
5. Bounds are configurable but have safe defaults — the system must not require explicit configuration to have bounds in place
6. The truncation indicator distinguishes bound-limited termination from successful completion and from failure — the caller/operator can determine that the run was truncated due to hitting a limit

## Acceptable Implementation Patterns

- Agent loop with iteration counter, checked at the start of each iteration, with configurable max_iterations defaulting to a safe value
- Tool call counter incremented at each invocation, checked before each call, with a configurable max_tool_calls limit
- Token budget tracker that accumulates token usage across LLM calls and enforces a ceiling
- Wall-clock timeout as a supplementary bound (not a replacement for iteration/call/token bounds)
- Run configuration object that carries all bounds and is checked by the orchestration layer
- Combination of multiple bounds: the run terminates when ANY bound is reached (not requiring all bounds to be hit)

## Evidence to Seek

- **Runtime evidence:** A run that hits a bound and terminates with a truncation indicator showing which bound was reached
- **Trace evidence:** Follow the agent loop and identify where bounds are checked — are they checked on every iteration? Before every tool call? After every LLM call?
- **Implementation evidence:** Iteration counter, tool call counter, token budget tracker, bound-checking code, truncation handling code, default values for bounds
- **Declarative evidence:** Configuration schema or documentation showing bound parameters and their defaults

## Trace Method

1. Find the agent loop (the iterative execution path where the LLM reasons, selects tools, and decides to continue)
2. Identify the loop termination conditions — what causes the loop to stop?
3. Check for iteration bounds: is there a counter? Is it checked? What is the maximum? Is there a default?
4. Check for tool call bounds: is there a tool invocation counter? Is it checked before each call?
5. Check for token spend bounds: is token usage tracked? Is there a ceiling?
6. Check the termination behavior when a bound is reached: does the run produce a truncation indicator? Can the caller distinguish truncation from completion?
7. Check for default values: if no configuration is provided, are bounds still in place?
8. Look for bypass paths: are there code paths that skip bound checking (e.g., "internal" tool calls that are not counted)?

## Verdict Conditions

**PASS when:** At least iteration and tool call bounds are defined and enforced with safe defaults AND bound-limited termination produces a clear truncation indicator distinguishable from success and failure AND bounds are checked consistently (no bypass paths).

**PARTIAL when:** Some bounds exist but not all critical ones (e.g., iteration bound but no tool call bound), OR bounds exist but have no safe default (require explicit configuration), OR truncation produces a generic failure rather than a distinct truncation indicator.

**GAP when:** No execution bounds exist — the agent loop can iterate indefinitely, OR bounds are defined in configuration but not enforced in code, OR the only termination condition is the LLM deciding to stop.

**UNASSESSABLE when:** The agent loop is implemented in a framework whose bound enforcement cannot be verified from the application code (e.g., bounds are configured but enforcement is in framework internals not in the codebase).

## Behavioral Verification Required?

**Required:** YES
**Test case:** Construct a test that deliberately exhausts each limit and verify the run completes with a truncation indicator rather than failing or looping indefinitely. Specifically:
- Set max_iterations to a low value (e.g., 2) and provide a task that would require more iterations — verify the run terminates with a truncation indicator
- Set max_tool_calls to a low value (e.g., 1) and provide a task that would invoke multiple tools — verify the run terminates with a truncation indicator
- Set max_tokens to a low value and provide a task that would consume more tokens — verify the run terminates with a truncation indicator
- For each test, verify that the run outcome distinguishes truncation from success and from failure

**What cannot be verified statically:** Whether bounds are actually enforced under real execution conditions. Static analysis can confirm that bound-checking code exists, but cannot confirm that it executes on every iteration, that counters are incremented correctly, or that the truncation path produces a clean outcome. Behavioral verification is required because unbounded execution is a CRITICAL severity issue and static analysis alone leaves residual risk.

## Boundary Notes

- P7.3 assesses execution bounds, not the quality of the agent's reasoning or its ability to complete tasks within bounds. A system that enforces bounds but frequently truncates runs due to low limits is a configuration issue, not a P7.3 gap.
- Wall-clock timeouts are a supplementary bound, not a replacement. A timeout prevents indefinite execution but does not prevent an agent from making hundreds of tool calls or consuming excessive tokens within the timeout window.
- Token bounds should cover total token consumption (input + output across all LLM calls in the run), not just a single LLM call's token limit. Per-call limits are set by the LLM provider; P7.3 assesses system-level bounds.
- "The LLM decided to stop" is not an execution bound. LLM termination decisions are unreliable and model-dependent. P7.3 requires bounds that are enforced by the system regardless of LLM behavior.
