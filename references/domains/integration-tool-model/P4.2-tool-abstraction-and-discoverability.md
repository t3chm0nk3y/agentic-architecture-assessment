## Intent

Tools must be registered and discoverable — not scattered or hardcoded. This applies regardless of the tool implementation pattern used (protocol-based servers, function registries, plugin systems, decorator-based registration, or any other approach). A consumer of the tool layer must be able to enumerate available tools and their interfaces without reading the implementation of every module.

## Why It Matters

In an agentic system, tools are the primary mechanism through which the agent acts on the world. When tools are hardcoded — wired directly into orchestration logic through conditional branches or inline function calls — adding, removing, or modifying a tool requires changes to the orchestration layer itself. This creates tight coupling between what the agent can do and how the agent decides what to do.

A discoverable tool registry decouples tool capabilities from orchestration logic. It enables dynamic tool selection, supports tool-level access control, and makes the system's capabilities enumerable for auditing and documentation. Without it, the answer to "what can this agent do?" requires reading the entire codebase.

## Applicability

Applies to: All systems that expose capabilities as callable tools, functions, or actions
Requires: At least two distinct tool capabilities

## N/A Conditions

N/A when the system has exactly one hardcoded action that it always performs. In that case, there is no tool selection problem and no need for a registry. However, this is rare in agentic systems.

## Required Behaviors

1. Tools are defined with explicit interfaces (input schema, output schema, description, or equivalent metadata)
2. Tools are registered in a discoverable location — a registry, catalog, manifest, or equivalent structure
3. The orchestration or agent layer discovers tools through the registry rather than importing them directly
4. Adding a new tool does not require modifying the orchestration layer
5. Available tools can be enumerated at runtime or through static inspection of the registry

## Acceptable Implementation Patterns

- Tool registry object that tools register themselves with via decorators, registration calls, or configuration
- MCP server exposing tool definitions through the standard tool listing protocol
- Function calling schema array passed to an LLM provider, assembled from a tool catalog
- Plugin directory where each plugin module exports a standard tool interface and is auto-discovered
- Tool manifest file (JSON, YAML, or equivalent) that declares available tools and their schemas
- Decorator-based registration where annotated functions are collected into a registry at startup

## Evidence to Seek

- **Runtime evidence:** A tool listing endpoint, CLI command, or log output that enumerates available tools and their metadata
- **Trace evidence:** Follow the path from tool definition to tool registration to tool invocation, verifying the registry mediates access
- **Implementation evidence:** Tool registry class or module, tool registration code, tool enumeration code, absence of hardcoded tool references in the orchestrator
- **Declarative evidence:** Tool manifest files, tool schema definitions, API documentation listing tools

## Trace Method

1. Find where tools are defined (function definitions, class definitions, configuration entries)
2. Check whether tools carry metadata (name, description, input/output schema)
3. Find the registration mechanism — are tools collected into a registry, catalog, or manifest?
4. Find the orchestration or agent layer that invokes tools
5. Check whether the orchestrator references tools by name through the registry or imports and calls them directly
6. Verify that adding a hypothetical new tool would only require defining the tool and registering it, not modifying the orchestrator
7. Check whether the set of available tools can be enumerated (list endpoint, registry query, manifest inspection)

## Verdict Conditions

**PASS when:** Tools are defined with metadata AND registered in a discoverable catalog AND the orchestration layer accesses tools through the catalog AND adding a tool does not require orchestrator changes.

**PARTIAL when:** Tools are registered but the registry is incomplete (some tools are registered, others are hardcoded), OR tools are registered but the orchestrator still contains hardcoded references to specific tools alongside registry-based access.

**GAP when:** No tool registry or catalog exists AND tools are hardcoded directly in the orchestration layer AND adding a tool requires modifying orchestration code.

**UNASSESSABLE when:** The tool mechanism is provided by a framework whose internal registration cannot be inspected from the codebase (e.g., tools are configured through an external admin interface not represented in code).

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether tools are actually discovered at runtime from the registry versus being loaded through some other mechanism. However, the presence of a registry, registration code, and registry-based access in the orchestrator is sufficient for static assessment.

## Boundary Notes

- This principle is technology-neutral. MCP servers, LangChain tools, OpenAI function calling schemas, plain function registries, plugin systems, and tool decorators are all equivalent patterns. The assessment looks for the behavior (registration and discoverability), not the specific mechanism.
- Internal helper functions used by the runtime are not "tools" for the purpose of P4.2. Tools are externally-facing capabilities that the agent selects and invokes to accomplish goals.
- A system may have multiple tool registries for different purposes (e.g., one for LLM-facing tools, one for system tools). Each registry must independently satisfy P4.2.
- Auto-discovery (scanning a directory for tool modules) is a valid registration mechanism as long as the discovered tools are enumerable after discovery.
