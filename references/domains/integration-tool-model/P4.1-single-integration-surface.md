## Intent

All external system interactions must go through a single, identifiable abstraction layer. No direct HTTP calls, database queries, or subprocess calls from the runtime or orchestration layers.

## Why It Matters

When external calls are scattered throughout the codebase, every module that touches an external system becomes a coupling point. A change to a vendor API, a database schema migration, or a subprocess interface change can require modifications in unpredictable locations. Worse, direct external calls bypass any centralized error handling, retry logic, or observability instrumentation. The result is a system where integration failures are discovered one at a time in production, each in a different module, each with different (or absent) error handling.

A single integration surface creates one place to enforce consistent patterns: retries, timeouts, circuit breaking, logging, normalization. It makes the system's external dependencies enumerable by reading the adapter layer rather than searching the entire codebase.

## Applicability

Applies to: All systems that interact with external services, databases, APIs, subprocesses, or file systems
Requires: At least one external dependency beyond the runtime itself

## N/A Conditions

None — any agentic system that performs actions in the world has external integrations. Even a system that only calls an LLM provider has at least one integration that should be abstracted.

## Required Behaviors

1. All external HTTP calls originate from an identifiable adapter or integration layer, not from runtime or orchestration code
2. All database queries originate from a data access layer or repository abstraction, not from business logic
3. All subprocess or shell invocations originate from a tool adapter, not from arbitrary runtime code
4. The adapter layer is structurally identifiable — through module organization, naming convention, or explicit registration
5. Runtime code depends on adapter interfaces (abstract types, protocols, contracts), not on concrete external clients

## Acceptable Implementation Patterns

- Dedicated adapter modules or packages that wrap external client libraries and expose internal interfaces
- Repository pattern for data access, where all database interactions go through repository methods
- Tool executor abstraction where all tool calls are dispatched through a single executor rather than invoked directly
- Gateway pattern where external API interactions are consolidated behind a gateway class or module
- Plugin architecture where each external integration is a self-contained plugin loaded through a common interface

## Evidence to Seek

- **Runtime evidence:** Execution traces showing all external calls originating from the adapter layer, with no direct external calls from runtime modules
- **Trace evidence:** Follow an external interaction from the point of invocation in the runtime through to the actual external call, verifying it passes through an adapter
- **Implementation evidence:** Module structure showing a clear adapter/integration layer; import analysis showing runtime modules depend on adapter interfaces, not external client libraries
- **Declarative evidence:** Architecture documentation describing the integration layer (lowest weight — verify against implementation)

## Trace Method

1. Identify the system's external dependencies (HTTP APIs, databases, LLM providers, file systems, subprocesses)
2. For each dependency, find the code that makes the actual external call (HTTP client invocation, database query execution, subprocess spawn)
3. Check whether that code resides in an identifiable adapter layer or is embedded in runtime/orchestration code
4. Search the runtime and orchestration layers for direct imports of external client libraries (HTTP libraries, database drivers, SDK clients)
5. If direct imports are found in the runtime, trace whether they are used for actual external calls or re-exported through an abstraction
6. Verify that the adapter layer exposes interfaces that the runtime depends on, rather than the runtime depending on concrete adapter implementations

## Verdict Conditions

**PASS when:** All identified external interactions go through an identifiable adapter layer AND runtime code does not contain direct external client calls AND the adapter layer is structurally distinct from the runtime.

**PARTIAL when:** An adapter layer exists and most external calls go through it, but some direct external calls exist in the runtime (e.g., a convenience HTTP call, a direct database query in a migration script, or a subprocess call outside the adapter).

**GAP when:** No adapter layer is identifiable AND external calls are made directly from runtime and orchestration code, scattered across multiple modules.

**UNASSESSABLE when:** The system uses a framework or dependency injection container that may provide implicit abstraction, but the abstraction cannot be confirmed from static analysis alone.

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether the adapter layer is actually the only path to external systems at runtime (e.g., monkey-patching or dynamic imports could bypass it). However, verifying structural separation and import dependencies is sufficient for static assessment.

## Boundary Notes

- This principle assesses whether external calls go through an abstraction layer, not whether the abstraction layer is well-designed (that is covered by P4.2, P4.3, and P4.5)
- Internal module-to-module communication is not governed by P4.1 — this is specifically about boundaries with external systems
- Configuration file reads and environment variable lookups are not considered external integrations for the purpose of P4.1, unless they involve network calls or external secret managers
- Test code that directly invokes external clients (e.g., integration test fixtures) does not violate P4.1 — the principle applies to production runtime code
