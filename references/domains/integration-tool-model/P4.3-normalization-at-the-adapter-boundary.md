## Intent

External data must be normalized to internal canonical schemas at the integration boundary — exactly once, in the adapter layer. Raw vendor payloads must not flow into the runtime. The adapter is the single point where external data shapes are translated into internal data shapes.

## Why It Matters

Every external system speaks its own dialect — different field names, different nesting structures, different conventions for nulls, dates, errors, and pagination. When raw vendor payloads propagate into the runtime, every module that touches that data must understand the vendor's schema. This creates invisible coupling: a vendor API change breaks not just the adapter, but every downstream consumer.

Normalizing at the adapter boundary means the vendor's schema is the adapter's problem. The runtime only sees canonical internal models. When a vendor changes their response format, exactly one adapter method changes. When a new vendor replaces an old one, only the adapter is rewritten — the runtime is untouched.

## Applicability

Applies to: All systems that consume data from external sources (APIs, databases, file systems, LLM providers, scraped content)
Requires: At least one external data source whose response format differs from the internal model

## N/A Conditions

N/A when the system's internal model is intentionally identical to a single external source's schema and there is no expectation of multi-vendor support. This is rare and fragile — it effectively means the system's internal model is owned by the external vendor.

## Required Behaviors

1. External response data is transformed into internal canonical types within the adapter layer before being returned to the runtime
2. Raw vendor response structures (JSON payloads, XML documents, protocol-specific wrappers) do not appear in runtime module type signatures or function parameters
3. The transformation from external to internal schema is explicit — implemented as mapping code, conversion functions, or model constructors, not implicit pass-through
4. Each external data source has its own normalization logic, isolated within its respective adapter
5. Error responses from external systems are normalized into internal error types at the adapter boundary

## Acceptable Implementation Patterns

- Adapter methods that accept internal request types and return internal response types, with vendor-specific serialization and deserialization internal to the adapter
- Mapping functions or conversion constructors that translate vendor response models to canonical domain models
- Response wrapper types in the adapter that parse vendor JSON into internal dataclasses, structs, or equivalent typed models
- Anti-corruption layer pattern where the adapter explicitly translates between the external bounded context and the internal bounded context
- Schema validation at the adapter boundary that rejects malformed external data before it enters the runtime

## Evidence to Seek

- **Runtime evidence:** Internal modules operating on canonical types that do not reference vendor-specific field names or structures
- **Trace evidence:** Follow external response data from the HTTP client (or equivalent) through the adapter transformation to the internal type returned to the runtime
- **Implementation evidence:** Mapping code in the adapter layer, canonical model definitions in a shared types module, type signatures on adapter public interfaces returning internal types
- **Declarative evidence:** Architecture documentation describing the normalization boundary (lowest weight)

## Trace Method

1. Identify an external integration (API call, database query, LLM invocation)
2. Find the raw response handling code — where is the external response first accessed?
3. Check whether the raw response is transformed into an internal type at that point or passed through
4. If transformed, verify the internal type is defined independently of the vendor schema
5. Follow the transformed data into the runtime — verify that runtime code references the internal type, not the raw vendor structure
6. Check error paths — are external error responses also normalized into internal error types at the adapter boundary?
7. Repeat for at least two different external integrations to confirm the pattern is consistent

## Verdict Conditions

**PASS when:** External response data is transformed into canonical internal types within the adapter layer AND raw vendor payloads do not appear in runtime type signatures AND the transformation is explicit and identifiable.

**PARTIAL when:** Normalization occurs for some integrations but not others, OR normalization is performed but raw vendor fields leak into the runtime through partial transformations (e.g., most fields are mapped but some vendor-specific fields pass through unchanged).

**GAP when:** Raw external response data flows directly into the runtime without transformation AND runtime modules reference vendor-specific fields or structures AND no normalization boundary is identifiable.

**UNASSESSABLE when:** The system uses auto-generated clients or SDK types that may or may not represent canonical models, and the distinction between vendor types and internal types cannot be determined from static analysis.

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether normalization handles all edge cases in vendor responses (e.g., null fields, unexpected types, version differences). However, the presence of explicit transformation code and canonical internal types is sufficient for static assessment.

## Boundary Notes

- P4.3 assesses whether normalization happens at the adapter boundary, not whether the canonical internal models are well-designed (that is P5.1)
- Pass-through adapters that simply re-export vendor types without transformation violate P4.3 even if they satisfy P4.1 structurally
- When an external system returns data that is already in the internal canonical shape (e.g., a service the team also owns), explicit mapping is still preferred but a documented passthrough with type assertion is acceptable
- LLM responses are external data and should be normalized at the adapter boundary. A raw LLM response string flowing into the runtime without parsing and validation is a P4.3 gap (and likely a P5.5 gap as well)
