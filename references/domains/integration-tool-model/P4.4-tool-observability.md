## Intent

Every tool invocation must be observable: inputs, outputs, latency, and errors must be captured. An operator must be able to determine, after the fact, which tools were called during an execution, what was passed to them, what they returned, how long they took, and whether they failed.

## Why It Matters

Tools are the primary side-effect mechanism in agentic systems. When something goes wrong — an incorrect action, an unexpected result, a timeout — the tool invocation is the first place to investigate. Without tool-level observability, debugging requires reproducing the entire execution to discover what each tool received and returned.

Tool observability also enables performance monitoring (which tools are slow?), cost tracking (which tools are called most often?), and safety auditing (what actions did the agent actually take?). In systems where LLMs select tools dynamically, observability provides the critical link between "the LLM chose to call tool X" and "here is exactly what tool X did."

## Applicability

Applies to: All systems that invoke tools or external capabilities as part of agent execution
Requires: At least one tool invocation pathway

## N/A Conditions

None — any system with tool invocations benefits from tool observability. Even a single tool with a single invocation pattern should be observable.

## Required Behaviors

1. Every tool invocation produces a record that includes the tool name or identifier
2. Tool inputs (arguments, parameters) are captured in the invocation record
3. Tool outputs (return values, results) are captured in the invocation record
4. Tool invocation latency (duration from call start to completion) is captured
5. Tool errors (exceptions, error responses, timeouts) are captured with sufficient context for diagnosis
6. Tool invocation records are correlated with the parent execution (via trace ID, run ID, or equivalent)

## Acceptable Implementation Patterns

- Middleware or decorator that wraps tool invocations and automatically captures inputs, outputs, latency, and errors
- OpenTelemetry spans around tool invocations with input/output attributes and error status
- Structured log entries emitted before and after each tool invocation with serialized inputs and outputs
- Event emission for tool invocation lifecycle (tool_started, tool_completed, tool_failed) with full context
- Tool executor abstraction that instruments all invocations passing through it

## Evidence to Seek

- **Runtime evidence:** Log output or trace data from an execution showing tool invocation records with inputs, outputs, latency, and error information
- **Trace evidence:** Follow a tool invocation from the call site through instrumentation to the observability output, verifying all required fields are captured
- **Implementation evidence:** Instrumentation code around tool invocations (decorators, middleware, wrapper functions), tool invocation record schema definitions
- **Declarative evidence:** Documentation describing tool observability (lowest weight)

## Trace Method

1. Find the tool invocation pathway (tool executor, tool dispatch function, or individual tool call sites)
2. Check whether there is instrumentation at the invocation point (wrapping, middleware, or explicit logging)
3. Verify that tool name/identifier is captured in the instrumentation output
4. Verify that tool inputs are serialized and captured
5. Verify that tool outputs are serialized and captured
6. Check for latency measurement (timing around the tool call)
7. Check error handling in the instrumentation — are tool failures captured with context, or silently swallowed?
8. Verify that tool invocation records include the parent execution's trace/correlation ID

## Verdict Conditions

**PASS when:** Tool invocations are instrumented AND inputs, outputs, latency, and errors are captured AND records are correlated with the parent execution AND the instrumentation covers all tool invocation paths.

**PARTIAL when:** Instrumentation exists but is incomplete — for example, inputs are logged but outputs are not, or the happy path is instrumented but error paths are not captured, or some tools are instrumented but others bypass the instrumentation.

**GAP when:** No tool invocation instrumentation exists AND tool calls are fire-and-forget from an observability perspective AND operators cannot determine post-hoc what tools were called or what they did.

**UNASSESSABLE when:** Tool invocations are handled by a framework or platform that may provide built-in instrumentation, but the instrumentation cannot be verified from the codebase alone.

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether instrumentation is active in production (it could be disabled by configuration), and whether the captured data is actually queryable by operators. However, the presence of instrumentation code covering inputs, outputs, latency, and errors is sufficient for static assessment.

## Boundary Notes

- P4.4 assesses whether tool invocations are observable, not whether the observability data is stored, queried, or visualized (those are operational concerns)
- Sensitive tool inputs (credentials, personal data) may need to be redacted in observability output. The principle requires capture of inputs, but redaction of sensitive fields is an acceptable and expected practice.
- Tool observability may legitimately truncate very large inputs or outputs (e.g., file contents, large payloads). Truncation with a size indicator is acceptable; complete omission is not.
- Internal function calls within the runtime are not "tool invocations" for P4.4 purposes. This principle applies to external-facing tool calls that represent agent actions.
