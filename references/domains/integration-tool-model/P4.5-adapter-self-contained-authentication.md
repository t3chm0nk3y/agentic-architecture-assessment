## Intent

Each external integration must manage its own authentication credentials internally. Credentials must not be passed through the runtime. The adapter is responsible for loading, refreshing, and applying credentials — the runtime should not know or care how authentication works for any given external system.

## Why It Matters

When credentials flow through the runtime — passed as function parameters, stored in shared state, or threaded through the orchestration layer — every module in the call chain gains access to secrets it does not need. This expands the blast radius of any security vulnerability, makes credential rotation harder (every call site must be updated), and creates a risk of credentials appearing in logs, error messages, or serialized state.

Self-contained authentication means each adapter loads its own credentials from a secure source (environment variables, secret manager, vault), refreshes them as needed, and applies them to outgoing requests — all internally. The runtime asks the adapter to "do something" and has no knowledge of the authentication mechanism used.

## Applicability

Applies to: All integrations that require authentication (API keys, OAuth tokens, database credentials, service accounts)
Requires: At least one external integration that uses credentials

## N/A Conditions

N/A when no external integrations require authentication. For example, an adapter that reads from a public, unauthenticated API does not need self-contained authentication.

## Required Behaviors

1. Each adapter loads its own credentials from a secure configuration source (environment, secret manager, configuration file, vault)
2. Credentials are not accepted as parameters on adapter public method signatures
3. Credentials are not stored in or retrieved from shared runtime state (global variables, singleton configuration objects accessible to the runtime)
4. Token refresh logic (if applicable) is handled within the adapter, transparent to the caller
5. Credential-related errors are surfaced as adapter errors without exposing the credential values themselves

## Acceptable Implementation Patterns

- Adapter constructor or factory that loads credentials from environment variables or a secret manager during initialization
- Credential provider abstraction internal to the adapter that handles loading, caching, and refresh
- OAuth token refresh middleware within the adapter's HTTP client configuration
- Database connection pool configured within the adapter module with credentials loaded from a secure source
- Per-adapter configuration objects that encapsulate credential loading and are not shared with the runtime

## Evidence to Seek

- **Runtime evidence:** Execution traces showing no credential values in runtime log entries, function parameters, or shared state
- **Trace evidence:** Follow credential loading from the secure source through to the point of use, verifying the entire path stays within the adapter
- **Implementation evidence:** Credential loading code inside adapter modules, absence of credential parameters on adapter public interfaces, adapter-internal token refresh logic
- **Declarative evidence:** Configuration documentation describing per-adapter credential management (lowest weight)

## Trace Method

1. Identify adapters that interact with authenticated external services
2. For each adapter, find where credentials are loaded (environment variable reads, secret manager calls, configuration file parsing)
3. Verify that credential loading occurs within the adapter module, not in the runtime or orchestration layer
4. Check adapter public method signatures — do any accept credential parameters (API keys, tokens, passwords)?
5. Check whether credentials are stored in shared state accessible to the runtime (global config objects, service locators, shared context)
6. If the integration uses token refresh (OAuth, rotating keys), verify that refresh logic is within the adapter
7. Check error handling — do credential errors expose credential values in error messages or log entries?

## Verdict Conditions

**PASS when:** Each adapter loads its own credentials internally AND credentials do not appear in adapter public method signatures AND token refresh (if applicable) is handled within the adapter AND credential values are not exposed in error messages.

**PARTIAL when:** Most adapters manage credentials internally, but some credentials are passed through the runtime (e.g., one adapter loads its own API key but another receives a token as a parameter), OR credential loading is internal but token refresh is handled externally.

**GAP when:** Credentials are passed through the runtime as function parameters, stored in shared global state, or configured in a central location that the runtime passes to adapters at call time.

**UNASSESSABLE when:** Credential management is handled by a framework or platform layer that is not visible in the codebase (e.g., credentials injected by a service mesh, cloud platform, or container orchestrator).

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether credentials are actually secure at runtime (e.g., environment variables could be logged by the deployment platform). However, verifying that credential loading is encapsulated within adapters and not passed through the runtime is sufficient for static assessment.

## Boundary Notes

- P4.5 assesses credential encapsulation within adapters, not the security of the credential storage mechanism itself (that is an infrastructure concern)
- A shared credential provider service that adapters call to obtain their credentials is acceptable as long as the runtime does not interact with the credential provider — only adapters do
- Test code that passes mock credentials to adapters for testing purposes does not violate P4.5. The principle applies to production runtime credential flow.
- Credentials passed in the original user request (e.g., a user-provided API key for a specific integration) are a design decision outside the scope of P4.5. P4.5 governs system-owned credentials for system integrations, not user-provided credentials for user-initiated actions.
