## Intent

Behavioral variation must be achievable through configuration changes, not code changes. Parameters that control how the agent behaves — model selection, retry strategies, guardrail thresholds, prompt selection, tool availability — must be externalized into configuration rather than hard-coded in source.

## Why It Matters

When behavioral parameters are hard-coded, every variation requires a code change, a review, a test cycle, and a deployment. This is disproportionate for changes that are operational in nature rather than architectural. Operators need to tune agent behavior (switch models, adjust timeouts, modify guardrail limits) without developer intervention. Configuration-driven behavior enables operational agility and separates "what the system can do" (code) from "how the system should behave in this context" (configuration).

## Applicability

Applies to: All agent systems with behavioral parameters
Requires: At least one parameter that varies between environments, contexts, or operational conditions

## N/A Conditions

None — all agentic systems have behavioral parameters that should be configurable. Even minimal systems benefit from externalizing at least model selection and timeout values.

## Required Behaviors

1. Behavioral parameters (model, temperature, max steps, timeout, retry count) are read from configuration, not hard-coded
2. Changing a behavioral parameter does not require modifying source code
3. Configuration values are loaded at startup or execution time from external sources (files, environment, service)
4. Default values exist for all behavioral parameters (the system functions without explicit configuration)
5. The system does not contain magic constants that control behavior without external override capability

## Acceptable Implementation Patterns

- Configuration files (YAML, TOML, JSON) loaded at startup with environment-specific overrides
- Environment variable-based configuration with typed parsing and defaults
- Layered configuration (defaults, file, environment, runtime override) with clear precedence
- Feature flags or behavioral toggles managed through a configuration service
- Per-execution configuration passed at invocation time that overrides defaults

## Evidence to Seek

- **Runtime evidence:** Changing a configuration value and observing different behavior without code changes
- **Trace evidence:** Follow a behavioral parameter from its use site in the runtime back to its configuration source
- **Implementation evidence:** Configuration loading code, parameter binding, absence of hard-coded behavioral constants
- **Declarative evidence:** Configuration documentation, example configuration files, configuration schema

## Trace Method

1. Identify behavioral parameters in the runtime (model name, temperature, max steps, timeouts, retry counts)
2. For each parameter, check its source: is it read from configuration or hard-coded?
3. Search for magic constants (literal numbers or strings that control behavior)
4. Check whether configuration can be changed without modifying source files
5. Verify that defaults exist for configurable parameters
6. Check for at least one parameter that varies between environments (dev/staging/production) as evidence of externalization

## Verdict Conditions

**PASS when:** All significant behavioral parameters are externalized to configuration. Changing behavior (model, thresholds, limits) requires only configuration changes, not code changes. Defaults exist for all parameters.

**PARTIAL when:** Some behavioral parameters are configurable but others are hard-coded. For example: model selection is configurable but retry counts or timeout values are magic constants in source.

**GAP when:** Behavioral parameters are predominantly hard-coded. Changing how the agent behaves requires modifying source code.

**UNASSESSABLE when:** Configuration loading exists but the mapping between configuration keys and behavioral parameters cannot be traced (e.g., configuration is loaded but consumed through a dynamic lookup mechanism that cannot be statically analyzed).

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether configuration changes actually take effect at runtime (could be overridden or ignored). However, tracing parameter sources to configuration loading code is sufficient for static assessment.

## Boundary Notes

- P8.4 assesses whether behavioral variation is configuration-driven; it does not assess the quality of the configuration system itself (that is Domain 5)
- Deployment configuration (database URLs, API keys, port numbers) is not the focus of P8.4 — the principle specifically targets parameters that control agent behavior
- A system where every parameter is configurable but the configuration has no defaults is still PASS for P8.4 (the default requirement prevents the system from being inoperable without configuration, but sparse defaults are acceptable)
- Hot-reload of configuration is not required; the test is whether changing configuration avoids code modification, not whether it avoids restart
