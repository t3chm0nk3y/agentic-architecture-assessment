## Intent

New workflows must be addable to the system without modifying the runtime or execution engine. The runtime must consume workflow definitions rather than encode workflow logic directly.

## Why It Matters

When adding a new workflow requires modifying the runtime, every addition carries regression risk. The execution engine becomes a bottleneck — both technically (merge conflicts, testing burden) and organizationally (only developers who understand the runtime can add workflows). A system where workflows are defined externally and loaded by the runtime scales linearly: the Nth workflow is as easy to add as the first.

## Applicability

Applies to: Systems with a workflow or multi-step execution concept
Requires: At least two distinct execution patterns or workflow types

## N/A Conditions

N/A when the system has no workflow concept — single-turn agents, stateless tool dispatchers, or systems where every execution follows the same fixed sequence. If there is only one execution pattern and no mechanism to define alternatives, workflow extensibility is not applicable.

## Required Behaviors

1. The runtime accepts workflow definitions from an external source (file, registry, configuration)
2. New workflow types can be introduced without modifying the runtime's source code
3. The runtime does not contain conditional branches that select behavior based on workflow type identifiers (no switch/case on workflow name)
4. Workflow definitions are loaded or registered through a consistent mechanism
5. Existing workflows continue to function when new workflows are added

## Acceptable Implementation Patterns

- Workflow definitions as declarative files (YAML, JSON, DSL) loaded by a generic executor
- Workflow registry with a registration interface that accepts new workflow definitions at startup or runtime
- Plugin-based workflow loading through directory scanning or dynamic imports
- Graph-based workflow definitions where the runtime is a generic graph executor
- State machine definitions consumed by a generic state machine runtime

## Evidence to Seek

- **Runtime evidence:** Adding a new workflow definition and verifying the runtime executes it without code changes
- **Trace evidence:** Follow a workflow from its definition source through loading/registration to execution, confirming the runtime is generic
- **Implementation evidence:** Workflow loader or registry code, absence of workflow-specific branches in the runtime
- **Declarative evidence:** Documentation describing how to add new workflows

## Trace Method

1. Identify all workflow types the system supports
2. Find where workflow behavior is defined — in external files/configs or inside the runtime
3. Check the runtime execution path for workflow-type-specific conditional logic
4. Simulate adding a new workflow: determine which files must change
5. If only definition files must change (not runtime code), the system is extensible
6. If the runtime requires modification, identify what specifically must change

## Verdict Conditions

**PASS when:** New workflows can be added by creating or registering a definition without modifying the runtime. Confirmed by identifying at least one workflow loaded through the extension mechanism and verifying the runtime has no workflow-type-specific branching.

**PARTIAL when:** Some workflows are loaded through an extension mechanism, but others are hard-coded in the runtime. Or a workflow registry exists but adding to it requires modifying runtime code (e.g., manually adding entries to a hard-coded list).

**GAP when:** All workflow behavior is encoded directly in the runtime. Adding a new workflow type requires modifying the execution engine's source code.

**UNASSESSABLE when:** The system has a workflow concept but the loading/registration mechanism cannot be determined from static analysis (e.g., relies on a framework's internal plugin system with no visible registration code).

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether a newly added workflow definition would actually be discovered and executed correctly by the runtime. However, tracing the loading mechanism and verifying the absence of workflow-specific branching in the runtime is sufficient for static assessment.

## Boundary Notes

- P8.1 assesses whether new workflows can be added without runtime modification; it does not assess the quality or expressiveness of the workflow definition format
- A system with only one workflow type is N/A for P8.1, not a GAP — the principle applies when the system has or intends to have multiple workflow types
- A workflow registry that requires restart to pick up new definitions is still extensible (hot-reload is not required) — the test is whether runtime source code must change, not whether a restart is needed
