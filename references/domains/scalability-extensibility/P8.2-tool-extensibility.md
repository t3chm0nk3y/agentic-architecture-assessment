## Intent

New tools must be addable to the system without modifying the agent loop. The system must maintain a dynamic tool catalog that the agent loop consumes generically, rather than having tool-specific logic embedded in the execution engine.

## Why It Matters

Tools are the primary mechanism through which agents interact with the world. A system where adding a new tool requires modifying the agent loop creates a bottleneck that limits the system's growth. Every tool addition becomes a change to the most critical and sensitive code path in the system. A dynamic tool catalog decouples tool growth from runtime stability — new tools are registered, discovered, and invoked through a generic interface that the agent loop does not need to know about in advance.

## Applicability

Applies to: All agent systems that invoke tools
Requires: A tool invocation mechanism of any kind

## N/A Conditions

None — any agentic system that invokes tools should support adding new ones without modifying the core execution loop. Even systems with a fixed tool set benefit from this separation for maintainability.

## Required Behaviors

1. A tool catalog or registry exists that the agent loop reads from at execution time
2. New tools can be added to the catalog without modifying the agent loop's source code
3. The agent loop dispatches tool invocations generically (by name, identifier, or interface) rather than through tool-specific branches
4. Tool definitions (name, description, schema, implementation) are decoupled from the agent loop
5. The agent loop does not contain conditional logic that checks for specific tool names or types

## Acceptable Implementation Patterns

- Tool registry with a registration interface that accepts new tool definitions
- Directory-based tool discovery (scanning a tools directory for conforming modules)
- Configuration-driven tool loading from tool definition files
- Decorator or annotation-based tool registration where adding a decorator to a function registers it as a tool
- Dynamic tool catalog populated at startup from a plugin system

## Evidence to Seek

- **Runtime evidence:** Adding a new tool definition and verifying the agent can discover and invoke it without code changes to the loop
- **Trace evidence:** Follow a tool invocation from the agent loop through dispatch to execution, confirming generic dispatch (no tool-specific branches)
- **Implementation evidence:** Tool registry/catalog code, tool registration mechanism, absence of tool-specific conditionals in the agent loop
- **Declarative evidence:** Documentation describing how to add new tools

## Trace Method

1. Find the agent loop or execution engine
2. Identify how tools are made available to the loop (hard-coded list, registry lookup, configuration)
3. Check the tool dispatch mechanism: does the loop dispatch by generic interface or by tool-specific branches?
4. Search the agent loop for tool-name-specific conditionals (if/switch on tool name)
5. Simulate adding a new tool: determine which files must change
6. If only a tool definition or registration call must be added (not agent loop code), the system is extensible

## Verdict Conditions

**PASS when:** New tools can be added by registering a definition without modifying the agent loop. Confirmed by identifying a dynamic tool catalog and verifying the agent loop dispatches generically without tool-specific branching.

**PARTIAL when:** A tool registry exists but the agent loop contains some tool-specific logic (e.g., special handling for certain built-in tools). Or new tools can be added to the registry but require a corresponding code change elsewhere (e.g., prompt template modification).

**GAP when:** Tool invocations are dispatched through tool-specific branches in the agent loop. Adding a new tool requires adding a new conditional branch to the execution engine.

**UNASSESSABLE when:** A tool registry exists but the dispatch mechanism between registry and execution cannot be traced statically (e.g., relies on framework-internal dispatch not visible in the codebase).

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether a newly registered tool would be correctly discovered by the LLM and invoked at runtime. However, verifying the registration mechanism and generic dispatch is sufficient for static assessment.

## Boundary Notes

- P8.2 assesses whether the agent loop requires modification to add tools; P4.2 (Tool Registration and Discovery) assesses whether tools are registered through a consistent mechanism. Both can be gaps independently — a system may have consistent registration but still require loop changes, or generic dispatch but inconsistent registration.
- Built-in tools that receive special treatment in the agent loop are a PARTIAL indicator, not necessarily a GAP — the key question is whether the extension path for new tools avoids the loop
- Hot-reloading of tools (adding tools without restart) is not required for PASS; the test is code modification, not restart avoidance
