## Intent

Workflow, skill, and tool definitions must be separated from the execution engine. Definitions describe what should happen; the execution engine determines how it happens. These must be distinct layers with a clear boundary between them.

## Why It Matters

When definitions are entangled with execution, the system cannot be extended, audited, or reasoned about at the definition layer alone. Operators cannot review what a workflow does without reading execution engine code. New definitions cannot be added without understanding the engine's internals. Testing requires running the full engine rather than validating definitions in isolation. Separation enables independent evolution — definitions can grow without engine changes, and the engine can be improved without definition changes.

## Applicability

Applies to: Systems with workflow, skill, or tool definitions that are consumed by an execution engine
Requires: At least one abstraction layer that separates "what to do" from "how to do it"

## N/A Conditions

N/A when the system has no definition layer — single-purpose agents with a fixed behavior that has no external definition, no workflow graph, no skill configuration, and no tool schema. If there is nothing to define externally, separation is not applicable.

## Required Behaviors

1. Definitions (workflows, skills, tools) exist as distinct artifacts from the execution engine code
2. The execution engine reads and interprets definitions rather than containing them inline
3. Definitions can be validated independently of the execution engine (schema validation, linting)
4. Changes to definitions do not require changes to the execution engine
5. Changes to the execution engine do not require changes to definitions (as long as the contract is maintained)
6. The boundary between definition and execution is explicit (clear loading/parsing/interpretation layer)

## Acceptable Implementation Patterns

- Declarative definition files (YAML, JSON, TOML) with a schema, loaded by a generic interpreter
- Definition objects constructed from external sources and passed to the engine as data
- A definition layer with its own validation and a contract interface consumed by the execution layer
- Separate packages or modules for definitions and execution with a defined API between them
- Template-based definitions that are rendered and interpreted by the engine

## Evidence to Seek

- **Runtime evidence:** Modifying a definition and verifying the engine behaves differently without engine code changes
- **Trace evidence:** Follow a definition from its source through loading and parsing to interpretation by the engine, confirming clear layer separation
- **Implementation evidence:** Definition files or modules distinct from engine modules, loader/parser code, definition schema or validation
- **Declarative evidence:** Architecture documentation describing the definition-execution boundary

## Trace Method

1. Identify the execution engine (runtime, orchestrator, agent loop)
2. Search for workflow definitions, skill definitions, and tool definitions
3. Check whether definitions are in separate files/modules from the engine
4. Check whether the engine has a loading or parsing step that reads definitions
5. Look for inline definitions (workflow graphs constructed inside the engine, skill parameters hard-coded in the loop, tool schemas built in dispatch code)
6. Verify that definitions have their own schema or validation independent of the engine

## Verdict Conditions

**PASS when:** Definitions are clearly separated from the execution engine. Definitions exist as distinct artifacts, the engine loads and interprets them, and changes to definitions do not require engine changes. Confirmed by identifying the loading boundary and verifying definitions are not constructed inline.

**PARTIAL when:** Some definitions are separated but others are inline. For example: tool definitions are external files but workflow definitions are hard-coded in the engine. Or definitions are in separate files but are imported and reconstructed by the engine rather than loaded through a generic mechanism.

**GAP when:** Definitions are entangled with the execution engine. Workflow logic, skill parameters, and tool schemas are constructed inside the engine's source code with no external definition layer.

**UNASSESSABLE when:** A definition layer exists but the boundary between definition and execution cannot be clearly identified (e.g., definitions and execution share modules with no clear separation of concerns).

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether definitions are fully self-describing or whether the engine requires implicit knowledge not present in the definition. However, verifying the structural separation and loading mechanism is sufficient for static assessment.

## Boundary Notes

- P8.5 is the foundational principle that P8.1, P8.2, and P8.3 depend on. A system can fail P8.5 while partially passing individual extensibility principles (e.g., tools are extensible but their definitions are constructed inline)
- "Separation" means a clear loading/interpretation boundary, not just file organization. Definitions in separate files that are imported and directly executed (not interpreted) are not truly separated
- The definition format (YAML vs. code vs. DSL) is not prescriptive — what matters is that definitions can be changed independently of the engine
- Systems that use code as their definition language (e.g., Python decorators defining tools) can still satisfy P8.5 if the decorated code is clearly a definition consumed by a generic engine, not execution logic itself
