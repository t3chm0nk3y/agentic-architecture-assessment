## Intent

New skills or personas must be addable to the system without modifying the runtime. Skills represent distinct behavioral configurations (tool sets, prompt strategies, constraints) that can be composed or selected at execution time, and the system must support their addition through definition rather than code change.

## Why It Matters

Skills and personas are the mechanism through which an agent's behavior is specialized for different tasks or contexts. When adding a new skill requires modifying the runtime, the system cannot scale its behavioral repertoire without accumulating risk. Each new skill becomes a code change to the core engine. A system where skills are defined externally and loaded by the runtime allows behavioral growth without architectural degradation — domain experts can define new skills without touching execution code.

## Applicability

Applies to: Systems with a skill, persona, role, or behavioral profile concept
Requires: At least one mechanism for varying agent behavior by context or task type

## N/A Conditions

N/A when the system has no skill or persona concept — agents that always behave identically regardless of context, with no mechanism to define or select behavioral variants. If the system has a single fixed behavior with no variation mechanism, skill extensibility is not applicable.

## Required Behaviors

1. The runtime accepts skill or persona definitions from an external source (file, registry, configuration)
2. New skills can be introduced without modifying the runtime's source code
3. Skill definitions specify behavioral parameters (tool access, prompt content, constraints) declaratively
4. The runtime loads and applies skill definitions through a consistent mechanism
5. Existing skills continue to function when new skills are added

## Acceptable Implementation Patterns

- Skill definitions as declarative files (YAML, JSON, Markdown with frontmatter) loaded by the runtime
- Skill registry with a registration interface that accepts new definitions
- Directory-based skill discovery (scanning a skills directory for conforming definitions)
- Skill composition through configuration where composite skills are assembled from primitives
- Profile-based behavioral selection where the runtime reads a profile definition and configures itself accordingly

## Evidence to Seek

- **Runtime evidence:** Adding a new skill definition and verifying the runtime can load and apply it without code changes
- **Trace evidence:** Follow a skill from its definition source through loading/registration to application at execution time
- **Implementation evidence:** Skill loader or registry code, skill definition schema, absence of skill-specific branches in the runtime
- **Declarative evidence:** Documentation describing how to add new skills or personas

## Trace Method

1. Identify all skills or personas the system supports
2. Find where skill behavior is defined — in external definitions or inside the runtime
3. Check how skills are loaded: file scanning, registry lookup, or hard-coded references
4. Check the runtime for skill-specific conditional logic (branches on skill name or type)
5. Simulate adding a new skill: determine which files must change
6. If only a definition file must be created (not runtime code modified), the system is extensible

## Verdict Conditions

**PASS when:** New skills can be added by creating a definition without modifying the runtime. Confirmed by identifying at least one skill loaded through the extension mechanism and verifying the runtime applies skill parameters generically.

**PARTIAL when:** Some skills are loaded through an extension mechanism, but others are hard-coded in the runtime. Or a skill registry exists but adding to it requires modifying runtime code.

**GAP when:** All skill behavior is encoded directly in the runtime. Adding a new skill or persona requires modifying the execution engine's source code.

**UNASSESSABLE when:** The system has a skill concept but the loading/application mechanism cannot be determined from static analysis.

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether a newly added skill definition would be correctly discovered, loaded, and applied by the runtime. However, tracing the loading mechanism and verifying generic application is sufficient for static assessment.

## Boundary Notes

- P8.3 assesses whether new skills can be added without runtime modification; it does not assess the quality or expressiveness of the skill definition format
- A system with no skill concept at all is N/A, not a GAP — the principle only applies when the system has or intends to have behavioral variants
- Skills that require code to implement their behavior (e.g., custom tool implementations) are still extensible under P8.3 if the skill definition and its association with the runtime are declarative — the tool code itself is covered by P8.2
