## Intent

All runtime configuration — timeouts, execution limits, connection strings, model names, server URLs, TTLs, retry counts, and similar operational parameters — must be externalized to configuration files or environment variables. No hardcoded values for these parameters should exist in runtime code.

## Why It Matters

Hardcoded configuration values create a tight coupling between code and environment. Changing a timeout, switching to a different model, or updating a service URL requires a code change, review cycle, and redeployment. In agentic systems, this is particularly damaging because these systems often have dozens of tunable parameters (LLM model identifiers, token limits, retry counts, circuit-breaker thresholds, cache TTLs) that operators need to adjust per environment, per deployment, or in response to incidents — without code changes.

Hardcoded values also resist discovery. When an operator asks "what is the current timeout for tool execution?", the answer should be in a configuration file, not buried in a source file that requires a code search to find.

## Applicability

Applies to: All system types
Requires: Any runtime parameter that varies by environment, deployment, or operator preference

## N/A Conditions

None — this principle applies to all agentic systems. Even minimal systems have at least one configurable parameter (e.g., a model name or endpoint URL).

## Required Behaviors

1. All timeout values are read from configuration, not hardcoded in source
2. All execution limits (max steps, max tokens, concurrency caps) are read from configuration
3. All connection strings and service URLs are read from configuration
4. All model identifiers (LLM model names, embedding model names) are read from configuration
5. All retry counts and backoff parameters are read from configuration
6. All TTL and cache expiration values are read from configuration
7. Configuration values have a single point of definition — not duplicated across files
8. Default values, when provided, are defined in the configuration layer (config file, schema defaults), not inline in consuming code

## Acceptable Implementation Patterns

- A typed settings object or config model that reads from environment variables and/or config files at initialization
- A configuration file (YAML, TOML, JSON) loaded at startup and injected into components
- Environment variable reading centralized in a config module, not scattered across runtime code
- A layered configuration system with explicit precedence (e.g., environment overrides file, file overrides defaults)
- Secrets management integration for sensitive values (API keys, connection strings)

## Evidence to Seek

- **Runtime evidence:** Configuration output or startup logs showing loaded values from external sources
- **Trace evidence:** Follow a specific parameter (e.g., a timeout value) from its point of use back to its definition; confirm it originates from a config source, not a code literal
- **Implementation evidence:** Search for hardcoded URL strings, integer timeout values, model name strings, and connection string patterns in non-config source files
- **Declarative evidence:** Configuration file templates, environment variable documentation, `.env.example` files

## Trace Method

1. Identify the configuration loading mechanism — look for config models, settings classes, or environment variable parsing modules
2. Search for hardcoded URL patterns in non-config source files (e.g., `http://`, `https://`, `localhost`, port numbers)
3. Search for hardcoded integer literals that appear to be timeouts, limits, or retry counts (e.g., `timeout=30`, `max_retries=3`, `limit=100`)
4. Search for hardcoded string literals that appear to be model identifiers (e.g., `"gpt-4"`, `"claude"`, model version strings)
5. Search for hardcoded connection strings or database URIs in non-config source files
6. For each hardcoded value found, determine whether it is a true configuration value (varies by environment) or a structural constant (does not vary)
7. Check whether default values are defined in the configuration layer or inline at consumption sites

## Verdict Conditions

**PASS when:** All runtime configuration parameters (timeouts, limits, URLs, model names, retry counts, TTLs) are read from externalized configuration AND no hardcoded operational values found in non-config source files AND defaults are defined in the configuration layer.

**PARTIAL when:** Configuration is externalized for some parameter categories but hardcoded values remain for others (e.g., URLs are externalized but timeouts are hardcoded), OR configuration is centralized but default values are defined inline at consumption sites rather than in the configuration layer.

**GAP when:** Configuration values are predominantly hardcoded in runtime source files, OR no configuration loading mechanism exists, OR the system relies entirely on inline defaults with no external override mechanism.

**UNASSESSABLE when:** The configuration loading mechanism exists but the actual configuration sources cannot be determined from the codebase alone (e.g., configuration is loaded from an external service whose contents are unknown).

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether externalized configuration values are actually used in production (operators might override them with environment variables not visible in the codebase). However, verifying that the code reads from configuration sources rather than hardcoded literals is sufficient for static assessment.

## Boundary Notes

- Structural constants that do not vary by environment (e.g., HTTP status codes, mathematical constants, protocol version strings) are not configuration and do not need to be externalized
- Test files may contain hardcoded values for test fixtures — this does not violate P2.1 (runtime code is the scope)
- Default values in a configuration schema or settings class are acceptable — the issue is defaults scattered across consuming code
- A value that is technically externalized but has a hardcoded fallback at every consumption site is effectively hardcoded; assess the actual behavior, not just the mechanism
