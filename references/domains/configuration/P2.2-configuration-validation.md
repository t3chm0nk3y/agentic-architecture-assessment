## Intent

Configuration must be validated against a schema or type system at startup. The system must not begin accepting work with invalid or incomplete configuration. Validation failures must surface immediately with clear, actionable diagnostic messages.

## Why It Matters

Externalized configuration (P2.1) shifts the source of truth from code to external files and environment variables — but this also shifts the failure mode. A typo in a config file, a missing environment variable, or a value of the wrong type will not be caught at compile time. Without validation at startup, these errors surface at runtime — often deep into an expensive execution, when an agent attempts to use a misconfigured value for the first time.

Startup validation converts these late, confusing failures into early, clear ones. An operator deploying with a missing API endpoint gets an immediate, specific error instead of a cryptic failure thirty minutes into an agent run.

## Applicability

Applies to: All system types that load configuration from external sources
Requires: Externalized configuration (see P2.1)

## N/A Conditions

If the system has no externalized configuration at all (all values are hardcoded), this principle is N/A — but that condition itself is a P2.1 gap.

## Required Behaviors

1. A configuration schema or typed model defines the expected structure, types, and constraints for all configuration values
2. Configuration is loaded and validated against this schema before the system begins accepting work
3. Missing required values produce a clear error message identifying the missing parameter
4. Type mismatches produce a clear error message identifying the parameter, expected type, and actual value
5. Constraint violations (e.g., negative timeout, empty required string, out-of-range value) produce a clear error message
6. Validation failure prevents the system from starting — the process exits or the initialization sequence halts
7. All validation errors are surfaced, not just the first one encountered (batch validation preferred)

## Acceptable Implementation Patterns

- A typed settings model (e.g., a dataclass, schema class, or typed configuration object) with built-in validation
- A schema validation step (JSON Schema, YAML schema) applied to the loaded configuration before initialization proceeds
- A dedicated configuration validation function called during the startup sequence that checks all required values and constraints
- A factory or builder pattern that constructs the configuration object and raises on invalid input
- Environment variable parsing libraries that enforce type and presence constraints at load time

## Evidence to Seek

- **Runtime evidence:** Startup logs or error output showing validation failure messages for intentionally invalid configuration
- **Trace evidence:** Follow the startup sequence from entry point to the point where the system begins accepting work; verify that configuration validation occurs between loading and work acceptance
- **Implementation evidence:** Configuration model/schema definition with type annotations and constraints, validation call in startup code, error handling for validation failures
- **Declarative evidence:** Documentation describing configuration validation behavior

## Trace Method

1. Find the configuration model or schema definition — look for typed config classes, settings models, or schema files
2. Check whether the model defines types, required fields, and value constraints (not just field names)
3. Find the startup sequence — the path from process entry to the system being ready to accept work
4. Verify that configuration loading and validation occur in this startup sequence, before work acceptance begins
5. Check what happens on validation failure — does the system halt, or does it continue with defaults or partial configuration?
6. Check the quality of error messages — do they identify the specific parameter, the expected value, and the actual value?
7. Check whether validation is batch (all errors reported) or fail-fast (stops at first error)

## Verdict Conditions

**PASS when:** A typed configuration model or schema exists AND is validated at startup before the system accepts work AND validation failure halts startup AND error messages are specific and actionable AND all required fields and constraints are covered.

**PARTIAL when:** A configuration model exists but is not validated at startup (model defined but validation never called), OR validation occurs but failure does not halt startup (system continues in a degraded state), OR validation covers types but not constraints (accepts negative timeouts, empty strings for required values).

**GAP when:** No configuration schema or typed model exists, OR configuration is loaded without any validation step, OR validation errors are silently swallowed and the system starts regardless.

**UNASSESSABLE when:** A configuration model exists and validation appears to be called, but the startup sequence is too complex to determine whether validation definitively precedes work acceptance (e.g., asynchronous initialization with unclear ordering).

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** The exact error messages produced at runtime for specific invalid inputs. However, the presence of a typed model, a validation call in the startup path, and halt-on-failure behavior can all be verified from code.

## Boundary Notes

- This principle assesses configuration validation specifically, not general input validation (request payloads, user input) — those are covered by schema and contract discipline
- A configuration model that exists but is only used for documentation (never instantiated or validated) does not satisfy this principle
- Runtime re-validation of configuration after hot-reload is a stronger signal but not required — startup validation is the minimum
- Default values in the configuration model are acceptable, but missing required values (those without defaults) must cause validation failure
