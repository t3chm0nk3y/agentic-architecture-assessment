## Intent

There must be one canonical configuration mechanism. Configuration must not be split across multiple sources without a clear precedence hierarchy. Behavioral parameters must not appear in both configuration and code.

## Why It Matters

When the same parameter is defined in multiple places — a config file, an environment variable, and a code constant — operators cannot determine which value is in effect without tracing the loading logic. Over time, values drift: one source is updated, the other is forgotten, and the system behaves according to whichever source happens to be read last. This creates silent configuration bugs that are extremely difficult to diagnose.

A single source of truth means that for any given parameter, there is exactly one canonical place to look. If multiple sources exist (e.g., config files with environment variable overrides), the precedence rules are explicit and documented, and no value is defined in more than one source at the same precedence level.

## Applicability

Applies to: All system types with externalized configuration
Requires: At least one configuration source (see P2.1)

## N/A Conditions

If the system has only a single configuration source (one config file, or only environment variables) with no possibility of duplication, this principle is trivially satisfied. It becomes meaningful when multiple configuration sources exist.

## Required Behaviors

1. There is one canonical configuration mechanism that the system reads from
2. If multiple configuration sources exist (e.g., file + environment variables), precedence rules are explicit and deterministic
3. The same parameter is not defined at the same precedence level in multiple sources
4. Configuration values do not appear as code constants that duplicate or shadow externalized values
5. Behavioral parameters (values that affect runtime behavior) are controlled through configuration, not through code constants that resist external override
6. The effective configuration (after all precedence resolution) is determinable without running the system

## Acceptable Implementation Patterns

- A single configuration file with no competing sources
- A layered configuration system with explicit, documented precedence (e.g., environment variables override file values, file values override defaults, defaults are defined in the schema)
- A configuration module that merges all sources in a defined order and exposes a single resolved configuration object
- A settings class that reads from a defined source hierarchy and resolves conflicts deterministically
- Configuration documentation that specifies source precedence and override rules

## Evidence to Seek

- **Runtime evidence:** The resolved configuration printed or logged at startup, showing values from their effective sources
- **Trace evidence:** Identify all configuration sources, then check for overlapping keys across sources; verify that precedence rules exist if overlap is found
- **Implementation evidence:** Configuration loading code showing source hierarchy, merge logic, and override rules; search for code constants that duplicate config values
- **Declarative evidence:** Documentation specifying configuration sources and precedence rules

## Trace Method

1. Identify all configuration sources — config files, environment variables, command-line arguments, code constants, database-stored settings
2. For each source, inventory the parameters it defines
3. Check for parameter overlap across sources — the same key or parameter defined in more than one source
4. For each overlap, check whether explicit precedence rules determine which source wins
5. Search for code constants (magic numbers, string literals in non-config source files) that duplicate or shadow externalized configuration values
6. Check whether the configuration loading code has a single merge/resolution point that produces one canonical configuration object
7. Assess whether an operator can determine the effective value of any parameter without running the system

## Verdict Conditions

**PASS when:** There is a single canonical configuration mechanism OR multiple sources with explicit, documented precedence rules AND no configuration values are duplicated across sources at the same precedence level AND no code constants shadow externalized configuration values.

**PARTIAL when:** Multiple configuration sources exist with implicit but consistent precedence (e.g., the code always reads environment first, then file, but this is not documented), OR a small number of configuration values are duplicated in code constants alongside their externalized definitions.

**GAP when:** Configuration values are scattered across multiple sources with no defined precedence, OR the same parameter is defined in multiple places with conflicting values, OR behavioral parameters are controlled by code constants that duplicate externalized configuration.

**UNASSESSABLE when:** The configuration loading mechanism uses a framework that resolves multiple sources automatically, and the precedence rules cannot be determined from the codebase alone (e.g., a configuration library with implicit resolution order that depends on runtime environment).

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** The actual effective configuration in a specific deployment (depends on which environment variables are set, which config files are present). However, the mechanism — whether sources are unified, whether precedence is defined, whether duplication exists — can be fully verified from code.

## Boundary Notes

- Multiple configuration files that are loaded and merged by a single mechanism with defined precedence are acceptable — "single source of truth" means single mechanism, not single file
- Environment variable overrides for config file values are a standard pattern and do not violate P2.4, provided the precedence is explicit
- Feature flags managed by an external service are a separate configuration source — assess whether they have a defined relationship to the primary configuration mechanism
- Test configuration that overrides production values for testing purposes does not violate P2.4 (test fixtures are outside the scope of runtime configuration discipline)
- The complexity of the precedence hierarchy itself is not assessed — a five-layer override system is acceptable if the precedence is explicit and documented
