## Intent

The system must refuse to start if critical dependencies are absent or misconfigured. This includes required external services being unreachable, referenced definitions being missing, required tool identifiers being absent, and invalid configuration values. Silent startup with degraded capability is a gap for required dependencies.

## Why It Matters

Agentic systems depend on external services (LLM providers, tool servers, data stores), internal definitions (workflow specifications, skill definitions, tool catalogs), and configuration correctness. When any of these critical dependencies are missing, the system will eventually fail — but it may not fail immediately. Instead, it starts successfully, accepts work, begins an expensive multi-step execution, and then crashes or produces incorrect results when it first attempts to use the missing dependency.

A startup safety gate converts these deferred failures into immediate ones. The system checks its critical invariants before accepting any work and refuses to start if they are not met. This is cheaper, clearer, and safer than discovering the problem mid-execution.

## Applicability

Applies to: Systems with a defined startup or initialization sequence
Requires: At least one external dependency or referenced definition

## N/A Conditions

This principle is N/A when the system has no startup sequence — for example, a notebook-based agent, a REPL-driven system, or a single-function invocation with no initialization phase. In these cases, there is no startup gate to enforce.

## Required Behaviors

1. The startup sequence identifies all critical dependencies (external services, referenced definitions, required tool identifiers, configuration prerequisites)
2. Each critical dependency is checked before the system begins accepting work
3. Unreachable required external services cause startup to halt with a specific error message
4. Missing referenced definitions (workflow specs, skill definitions, tool catalog entries) cause startup to halt with a specific error message
5. Missing required tool identifiers cause startup to halt with a specific error message
6. The system distinguishes between required and optional dependencies
7. Optional dependencies may degrade gracefully with a logged warning; required dependencies must halt startup on failure
8. Startup halt messages identify the specific missing or failing dependency, not just "startup failed"

## Acceptable Implementation Patterns

- Health check or readiness probes that verify external service connectivity before marking the system as ready
- A startup validation phase that iterates over declared dependencies and checks each one
- Dependency injection containers that fail construction if required services are unavailable
- A pre-flight check function called before the main loop or server listener starts
- Circuit breaker initialization that tests connectivity during startup
- Definition loaders that validate referenced identifiers against available catalogs at startup

## Evidence to Seek

- **Runtime evidence:** Startup failure output when a required dependency is intentionally removed or made unreachable
- **Trace evidence:** Follow the startup sequence from entry point to work acceptance; identify each dependency check point and what happens on failure
- **Implementation evidence:** Startup code that checks for external service availability, validates referenced definitions exist, and halts on failure
- **Declarative evidence:** Documentation describing required vs optional dependencies and startup behavior

## Trace Method

1. Find the startup or initialization sequence — the code path from process entry to the system being ready to accept work
2. Identify all external dependencies the system uses at runtime (service endpoints, databases, LLM providers, tool servers)
3. Check whether each dependency is verified during startup (connectivity check, health check, ping)
4. Identify all internal references the system resolves at runtime (workflow definitions, skill specifications, tool catalog entries)
5. Check whether each reference is validated during startup (existence check, schema validation)
6. For each dependency check, determine what happens on failure — does the system halt, log a warning and continue, or silently ignore?
7. Check whether required and optional dependencies are distinguished — required dependencies should halt, optional should warn
8. Verify that the system does not begin accepting work until all required dependency checks pass

## Verdict Conditions

**PASS when:** The startup sequence checks all critical dependencies AND halts with specific error messages when required dependencies are missing or unreachable AND distinguishes between required and optional dependencies AND all checks occur before work acceptance.

**PARTIAL when:** Some critical dependencies are checked but others are not (e.g., external service connectivity is checked but referenced definitions are not validated), OR dependency checks exist but failure does not halt startup (system logs a warning and continues for required dependencies), OR required and optional dependencies are not distinguished.

**GAP when:** No dependency checking occurs at startup, OR the system starts successfully with missing critical dependencies, OR startup failures are silently swallowed.

**UNASSESSABLE when:** The system has a startup sequence but the dependency initialization is handled by a framework or container whose behavior cannot be determined from the codebase alone (e.g., the system relies on an external orchestrator for dependency management).

## Behavioral Verification Required?

**Required:** Recommended but not strictly required
**What cannot be verified statically:** Whether the system actually halts in practice when a dependency is missing (the halt behavior may depend on exception propagation that is hard to trace statically). If the startup code clearly raises or exits on failure, static analysis is sufficient. If the failure handling is ambiguous, behavioral verification adds confidence.

## Boundary Notes

- This principle assesses startup behavior specifically, not runtime resilience — what happens when a dependency fails after startup is covered by error handling and graceful degradation principles
- The distinction between required and optional is critical: a system that treats all dependencies as optional and silently degrades is not passing P2.3 if any of those dependencies are functionally required
- Health checks or readiness probes that run after startup (e.g., Kubernetes readiness probes) are a positive signal but do not substitute for startup gating — the system should not begin accepting work until checks pass, regardless of external probing
- Startup performance is not assessed here — a startup sequence that takes longer because it validates dependencies is acceptable
