## Intent

Observability must be consistent across all modules — not concentrated in one layer while others are dark. Uneven coverage creates blind spots that degrade debugging and monitoring.

## Why It Matters

A system where the runtime is fully instrumented but the adapter layer has no tracing creates a gap in the execution timeline. When diagnosing latency, the time between "runtime started step" and "step completed" is a black box — the operator cannot tell if the delay is in the database, a tool call, or the LLM. Consistent coverage across modules means any performance issue or failure can be localized.

## Applicability

Applies to: Systems with more than one module or distinct component layer
Requires: At least two modules or layers with different responsibilities

## N/A Conditions

N/A when the system is a single-file script or single-module application with no distinct component layers. If there are no module boundaries, there is no uneven coverage to assess.

## Required Behaviors

1. Logging is present in all modules that participate in execution (runtime, adapters, API layer, etc.)
2. If distributed tracing is used, trace spans cover operations in all modules — not just the runtime
3. If event emission is used, events cover transitions across all modules — not just the orchestration layer
4. Observability gaps do not exist at module boundaries where execution crosses from one module to another
5. Adapter/integration operations (database queries, cache operations, external API calls) are instrumented with latency and outcome

## Acceptable Implementation Patterns

- Distributed tracing (OpenTelemetry, Logfire, Datadog) with spans in every module
- Consistent structured logging across all modules with shared log format and context
- Event emission at every module boundary crossing
- Middleware-based instrumentation that automatically covers operations in all layers
- Per-module instrumentation with shared trace context propagation

## Evidence to Seek

- **Runtime evidence:** Trace output showing spans from multiple modules in a single execution
- **Trace evidence:** Follow an execution from API endpoint through runtime to adapter to external call — check for instrumentation at each layer
- **Implementation evidence:** Logging/tracing setup in each module, span creation in adapter code, instrumentation in external call handlers
- **Declarative evidence:** Observability architecture documentation

## Trace Method

1. Identify all modules or component layers in the system
2. For each module, check for logging/tracing infrastructure (logger initialization, span creation, event emission)
3. Compare observability density across modules — is one module fully instrumented while another has only basic logging?
4. Check adapter/integration code specifically — external calls (database, cache, API) are common blind spots
5. If distributed tracing is used, check whether the runtime module has spans AND the adapter module has spans, or only the runtime
6. Look for asymmetry: if the runtime logs every operation but the database adapter only logs errors, that is uneven coverage

## Verdict Conditions

**PASS when:** All modules participating in execution have consistent observability coverage — logging, tracing, and/or event emission are present across all layers without significant gaps.

**PARTIAL when:** Observability exists in all critical modules but coverage is uneven — some modules are fully instrumented while others have only basic logging or are missing trace spans for key operations.

**GAP when:** One or more modules participating in execution have no observability instrumentation — operations in those modules are invisible to operators.

**UNASSESSABLE when:** The system uses a framework that may provide automatic instrumentation, but it cannot be determined statically which operations are covered.

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether automatic instrumentation (e.g., framework-provided tracing) actually covers all operations at runtime. However, checking for explicit instrumentation code across modules is sufficient for static assessment.

## Boundary Notes

- P3.6 is inherently a relative assessment — it compares coverage across modules rather than checking a binary condition. The assessor must judge what constitutes "consistent" coverage for the system's architecture.
- Not all modules require the same depth of instrumentation. A utility module with pure functions may need only error logging, while an adapter module making external calls needs full span instrumentation. The assessment should be proportional to the module's role in execution.
- Automatic instrumentation by frameworks (e.g., FastAPI auto-instrumentation by Logfire) counts as coverage for the modules it covers. The concern is modules that fall outside the automatic coverage.
- P3.6 does not require every function to be instrumented — it requires that every module boundary crossing and external operation is observable. Interior module operations can be instrumented at the discretion of the implementer.
