## Intent

All logging must be structured (key-value or JSON), not free-form strings. Log entries must include at minimum: timestamp, level, component, and run/trace ID where applicable.

## Why It Matters

Free-form log strings cannot be reliably searched, filtered, or aggregated. When an operator needs to find all log entries for a specific run, structured fields make this a single query. With free-form strings, it requires regex guessing across inconsistent formats. At scale, unstructured logs are functionally useless for debugging.

## Applicability

Applies to: All system types
Requires: Any form of logging output

## N/A Conditions

None — this principle applies to all agentic systems. Even minimal systems benefit from structured logging.

## Required Behaviors

1. A structured logging library or framework is configured and used
2. All log statements produce structured output (key-value pairs or JSON), not free-form strings
3. Log entries include at minimum: timestamp, log level, and component/module identifier
4. Log entries within an execution context include the trace/run identifier (see P3.1)
5. No `print()` statements, `console.log()` without structure, or stdlib `logging.info("string")` without context binding are used for operational logging

## Acceptable Implementation Patterns

- structlog (Python) with bound context loggers
- winston (Node.js) with JSON transport and default metadata
- zerolog or zap (Go) with structured field logging
- slog (Go 1.21+) with handler-based structured output
- Serilog (.NET) with structured templates
- log/slog with context propagation
- Any logging library that produces key-value or JSON output with context binding

## Evidence to Seek

- **Runtime evidence:** Sample log output showing JSON or key-value format with consistent fields
- **Trace evidence:** Follow logging setup from initialization through to call sites in different modules
- **Implementation evidence:** Logger configuration, context binding code, log call patterns across modules
- **Declarative evidence:** Logging configuration files or documentation

## Trace Method

1. Find the logging setup/initialization code (often in a startup module or config file)
2. Identify the logging library and its configuration (JSON output, context binding, log level)
3. Sample 3-5 log call sites across different modules — check for structured calls vs free-form strings
4. Search for `print(` or equivalent unstructured output in non-test source files
5. Check whether log calls bind execution context (run_id, correlation_id) or just emit standalone messages
6. If a structured library is configured, verify it is actually used at call sites (not bypassed with stdlib logging)

## Verdict Conditions

**PASS when:** A structured logging library is configured AND all sampled log call sites use structured output with context binding AND no `print()` or unstructured logging calls found in operational code.

**PARTIAL when:** A structured logging library is configured but some call sites use unstructured output, OR structured logging is used but without consistent context binding (trace IDs missing from log entries).

**GAP when:** No structured logging library is configured, OR `print()` / unstructured `logging.info("string")` is the primary logging mechanism.

**UNASSESSABLE when:** The logging library is configured but no log call sites could be found to verify actual usage patterns.

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** The actual format of log output in production (may depend on logging configuration that differs per environment). However, the logging library configuration and call site patterns are sufficient for static assessment.

## Boundary Notes

- Test files may use `print()` for debugging — this does not violate P3.2 (operational logging is the scope)
- Script-level utilities or one-off tools within a project may use simpler logging — assess the main system, not auxiliary scripts
- The specific logging library does not matter — what matters is structured output with context binding
