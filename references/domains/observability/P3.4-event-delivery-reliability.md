## Intent

If the system supports real-time event streaming (SSE, WebSocket, etc.), the stream must support reconnection without losing events. A client that disconnects and reconnects must be able to receive events it missed. This requires event buffering and sequence numbers.

## Why It Matters

Network interruptions are inevitable in production. If a monitoring dashboard, audit tool, or UI disconnects during a run, it must be able to reconnect and resume without missing events. Without reconnection support, clients display incomplete or stale run state after any network blip, eroding operator trust in the observability layer.

## Applicability

Applies to: Systems with real-time event streaming interfaces (SSE, WebSocket, message queues)
Requires: A streaming endpoint that delivers events to clients in real time

## N/A Conditions

N/A when the system has no real-time event streaming interface. Systems that only offer polling endpoints, synchronous request-response APIs, or batch event retrieval do not need reconnection support for a streaming protocol they do not offer.

## Required Behaviors

1. Events carry monotonically increasing sequence numbers or equivalent ordering tokens
2. Events are buffered for a defined window (configurable or with a reasonable default)
3. Clients can reconnect and request events from a specific sequence number or timestamp
4. The system delivers missed events on reconnection before streaming new events
5. Buffer retention is bounded (events are not retained indefinitely)

## Acceptable Implementation Patterns

- SSE with `Last-Event-ID` header support and server-side event buffer
- WebSocket with client-tracked sequence numbers and replay-on-reconnect protocol
- Event log in a database or message queue with consumer offset tracking (Kafka-style)
- Redis/Valkey-backed event buffer with per-stream sequence numbers and range queries
- In-memory ring buffer with sequence-based retrieval (acceptable for single-instance deployments)

## Evidence to Seek

- **Runtime evidence:** Reconnection test showing missed events delivered after reconnect
- **Trace evidence:** Follow event emission from producer through buffer to consumer, checking for sequence assignment and buffer writes
- **Implementation evidence:** Sequence number assignment on events, buffer write code, reconnection handler with sequence-based retrieval
- **Declarative evidence:** Streaming API documentation describing reconnection protocol

## Trace Method

1. Find the streaming endpoint (SSE, WebSocket, or equivalent)
2. Check whether events carry sequence numbers or ordering tokens
3. Find the event buffer (in-memory, cache-backed, or database-backed)
4. Check buffer write code — are events written to the buffer at emission time?
5. Check the reconnection handler — does it accept a sequence number or offset parameter?
6. Verify that reconnection retrieves events from the buffer starting at the requested sequence
7. Check buffer retention policy — is there a TTL or size limit?

## Verdict Conditions

**PASS when:** Events carry sequence numbers AND are buffered AND the streaming endpoint supports reconnection with sequence-based retrieval AND buffer retention is bounded.

**PARTIAL when:** Events carry sequence numbers but buffering or reconnection support is incomplete (e.g., events are sequenced but no buffer exists, or buffer exists but no reconnection protocol).

**GAP when:** The system offers real-time event streaming but has no reconnection support — disconnected clients lose events permanently.

**UNASSESSABLE when:** A streaming endpoint exists but the reconnection mechanism cannot be determined from static analysis (e.g., handled by a reverse proxy or framework middleware not visible in the codebase).

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether reconnection actually delivers missed events under real network conditions. However, the presence of sequence numbers, buffer writes, and reconnection handlers is sufficient for static assessment.

## Boundary Notes

- P3.4 only applies to real-time streaming interfaces. A system that stores events in a database and offers a polling API for retrieval does not need SSE reconnection support — the database IS the durable store.
- If the system uses a message broker (Kafka, RabbitMQ, etc.) for event delivery, the broker's built-in consumer offset tracking may satisfy P3.4 without custom reconnection logic.
- Buffer size and TTL are implementation choices, not principle requirements — as long as some reasonable bounded buffer exists.
