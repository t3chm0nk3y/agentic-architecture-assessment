## Intent

Every agent execution must have a unique identifier (run ID, trace ID, correlation ID) that is propagated across all components and external calls for the duration of that execution.

## Why It Matters

Without a propagated trace identifier, diagnosing failures in production requires correlating timestamps across unrelated log streams — a process that fails under concurrent load. When multiple runs execute simultaneously, interleaved logs become impossible to separate. Every downstream debugging tool (log search, trace visualization, event replay) depends on this identifier existing and being consistently propagated.

## Applicability

Applies to: All system types
Requires: Any form of execution that spans more than a single function call

## N/A Conditions

None — this principle applies to all agentic systems. Even single-file scripts benefit from a trace identifier when they log.

## Required Behaviors

1. A unique identifier is generated at the start of each execution
2. The identifier is propagated to all log entries produced during the execution
3. The identifier is propagated to all events emitted during the execution
4. The identifier is propagated to all external tool/service calls made during the execution
5. The identifier is included in all error/failure records produced during the execution
6. The identifier is available in any API response or event stream associated with the execution

## Acceptable Implementation Patterns

- Correlation ID middleware that generates an ID at the API boundary and binds it to a context-local logger
- OpenTelemetry trace context propagation with automatic span context inheritance
- Run ID generated at execution start and passed explicitly as a parameter through the call chain
- Context-local storage (Python contextvars, Go context.Context, Node.js AsyncLocalStorage) with a bound identifier
- Request-scoped dependency injection of a trace context object

## Evidence to Seek

- **Runtime evidence:** Log output from a real or test execution showing the same identifier across multiple log entries from different components
- **Trace evidence:** Follow the identifier from generation point through logging setup, event emission, tool invocation, and error handling
- **Implementation evidence:** Identifier generation code, context binding code, propagation mechanism
- **Declarative evidence:** Documentation or specs describing trace propagation (lowest weight)

## Trace Method

1. Find the execution entry point (API handler, CLI entry, main loop)
2. Search for ID generation: look for `uuid`, `trace_id`, `correlation_id`, `run_id`, `request_id` generation
3. Identify the propagation mechanism: context-local binding, parameter passing, or middleware injection
4. Trace to logging: verify the ID appears in logger context or is passed to log calls
5. Trace to events: verify the ID is a field on emitted events
6. Trace to tool/external calls: verify the ID is passed to integration calls (as a parameter, header, or context attribute)
7. Trace to error handling: verify the ID appears in error records or exception context

## Verdict Conditions

**PASS when:** A unique identifier is generated per execution AND propagated to logging, events, tool calls, and error handling. Confirmed by tracing the identifier through at least two module boundaries.

**PARTIAL when:** An identifier is generated but not propagated to all layers. For example: present in logs but missing from events, or present in the runtime but not propagated to tool calls.

**GAP when:** No execution-scoped identifier is generated, OR the identifier exists but is not propagated beyond the point of generation.

**UNASSESSABLE when:** The system has an identifier but the propagation mechanism cannot be determined from static analysis (e.g., relies on framework magic that requires runtime inspection).

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether the identifier actually appears in production log output (could be suppressed by logging configuration). However, tracing the propagation mechanism through code is sufficient for a static assessment.

## Boundary Notes

- This principle assesses identifier generation and propagation, not the format or quality of what is logged (that is P3.2)
- A system may use multiple identifiers for different purposes (run_id for business logic, trace_id for distributed tracing, correlation_id for request correlation) — any of these satisfies P3.1 as long as at least one is propagated end to end
- If the system uses a framework that automatically propagates trace context (e.g., OpenTelemetry), verify that the automatic propagation actually reaches all components — framework defaults may not cover custom code paths
