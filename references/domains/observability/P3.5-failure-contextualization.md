## Intent

All failures must be captured with structured diagnostic context: what failed, where, why, and what the execution state was at the time. Stack traces alone are insufficient.

## Why It Matters

When an agent execution fails in production, the first question is "what happened?" If the failure record contains only a stack trace or a generic error message, the operator must reconstruct context manually — reading logs, correlating timestamps, and guessing at the execution state. Structured failure context (which run, which step, which tool, what the input was, what the error class is) reduces mean time to diagnosis from hours to seconds.

## Applicability

Applies to: All system types
Requires: Any error handling mechanism

## N/A Conditions

None — this principle applies to all agentic systems. Every system that can fail must capture structured context about failures.

## Required Behaviors

1. Failures are captured with structured fields, not just messages or stack traces
2. Failure records include at minimum: error classification/type, human-readable message, and execution context (which run, step, or component)
3. Failure records include structured diagnostic detail (input that caused the failure, relevant state, etc.) beyond the error message
4. The trace/run identifier (P3.1) is present in failure records
5. Bare `except: pass`, `catch (e) {}`, or equivalent silent error swallowing does not occur in operational code paths

## Acceptable Implementation Patterns

- Typed error model with classification enum, message, detail dict, and context IDs
- Exception hierarchy with structured attributes (error_code, context, detail)
- Error event emission with structured payload at the point of failure
- Error logging with bound context (run_id, step_id, component) via structured logger
- Result types (Rust Result, Go error wrapping) with structured context

## Evidence to Seek

- **Runtime evidence:** Error output from a failed execution showing structured fields with execution context
- **Trace evidence:** Follow a failure path from origin through error handling to final error record — check what context is captured at each stage
- **Implementation evidence:** Error model/class definitions, exception handler code, error logging patterns
- **Declarative evidence:** Error model documentation

## Trace Method

1. Find the error model or error class definition (if one exists)
2. Check the error model for structured fields: classification, message, detail, context IDs
3. Find exception handlers in the execution path (try/catch blocks, error middleware)
4. Verify that exception handlers capture structured context — not just the exception message
5. Check for silent error swallowing: `except: pass`, empty catch blocks, errors caught but not recorded
6. Trace a failure from a tool invocation or LLM call through to the final error record — check what context survives

## Verdict Conditions

**PASS when:** Failures are captured with structured diagnostic context (classification, message, detail, execution context) AND the trace identifier is included in failure records AND no silent error swallowing observed in operational code paths.

**PARTIAL when:** Some failure context exists but is incomplete (e.g., error messages without classification, or classification without execution context), OR structured context is present in some modules but not others.

**GAP when:** Failures are captured only as generic messages or stack traces without structured context, OR silent error swallowing (bare except/catch) is found in critical code paths.

**UNASSESSABLE when:** Error handling code exists but the actual error output format cannot be determined without runtime inspection.

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** The actual error output in production. However, the error model structure and exception handler patterns are sufficient for static assessment.

## Boundary Notes

- P3.5 overlaps with Error Handling (Domain 6) but assesses observability specifically: can an operator understand what failed and why? P6.1 assesses the error model itself as a data structure.
- Silent error swallowing in non-critical paths (e.g., best-effort cache writes) may be acceptable if the swallowing is intentional, logged at a warning level, and does not affect execution correctness. The concern is errors in critical execution paths being silently discarded.
- Stack traces have value as supplementary detail but are insufficient as the sole diagnostic context. An error record with only a stack trace and no classification, execution context, or structured detail is a PARTIAL at best.
