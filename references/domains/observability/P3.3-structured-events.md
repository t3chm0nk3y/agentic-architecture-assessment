## Intent

Significant lifecycle transitions (execution start/end, step start/end, tool invocations, failures) must emit structured events. Clients must be able to reconstruct full run state from the event stream alone — if this is not possible, the event model is incomplete.

## Why It Matters

Structured events are the primary mechanism for real-time observability and post-hoc auditing. Without them, clients (UIs, monitoring systems, audit tools) must poll for state changes or parse logs. A complete event stream enables: real-time run visualization, automated alerting on specific transitions, replay of execution history, and verification that the system behaved as expected.

## Applicability

Applies to: Systems with multi-step execution, persistent run state, or streaming interfaces
Requires: At least two distinct lifecycle transitions (e.g., run start and run end)

## N/A Conditions

N/A when the system is a synchronous, single-turn agent with no persistent run state, no streaming interface, and no multi-step execution. In such systems, the request-response cycle itself is the event.

## Required Behaviors

1. Lifecycle transitions emit structured events with consistent schema
2. At minimum, these transitions emit events: execution start, execution end (success or failure), and any intermediate state changes
3. Each event includes: event type, timestamp, execution identifier, and relevant payload
4. If the system has steps or sub-operations, each step start/end emits an event
5. If the system invokes tools, each tool invocation and result emits an event
6. The event stream is complete enough that a consumer could reconstruct the run's lifecycle without additional data sources
7. All defined event types are actually emitted (dead event types that are defined but never emitted are a gap in coverage)

## Acceptable Implementation Patterns

- Typed event bus with publish/subscribe and serialized event objects
- SSE endpoint streaming JSON events per lifecycle transition
- WebSocket channel with typed message frames
- Event log written to a database or message queue with structured records
- In-process event emitter with typed handlers (e.g., Node.js EventEmitter with typed events)
- Callback-based hooks that receive structured event payloads

## Evidence to Seek

- **Runtime evidence:** Event stream output showing structured events at lifecycle transitions
- **Trace evidence:** Follow an execution from start to end, checking for event emission at each transition
- **Implementation evidence:** Event type definitions, event emission call sites, event schema/model
- **Declarative evidence:** Event type documentation or streaming API specifications

## Trace Method

1. Find the event type definitions (enum, constants, or type discriminator)
2. For each defined event type, search for emission sites in the codebase
3. Verify that event types cover the full lifecycle: start, end, intermediate transitions, failures
4. Trace one complete execution path and check for event emission at each state transition
5. Check for defined-but-never-emitted event types (dead event types)
6. Verify that events include execution context (run ID, step ID) and a timestamp

## Verdict Conditions

**PASS when:** All significant lifecycle transitions emit structured events AND all defined event types have at least one emission site AND events include consistent schema with execution context.

**PARTIAL when:** Events are emitted for some transitions but not all (e.g., run events exist but step events are missing), OR some defined event types are never emitted (dead event types), OR events exist but lack consistent schema or execution context.

**GAP when:** No structured event emission exists — lifecycle transitions produce only log lines or nothing observable.

**UNASSESSABLE when:** Event types are defined but emission sites cannot be traced (e.g., emitted through a framework mechanism that cannot be statically analyzed).

## Behavioral Verification Required?

**Required:** No
**What cannot be verified statically:** Whether events are actually delivered to consumers at runtime. However, verifying that emission code exists at each lifecycle transition is sufficient for static assessment.

## Boundary Notes

- P3.3 assesses whether events are emitted at lifecycle transitions; P3.4 assesses delivery reliability. A system may emit events correctly (P3.3 PASS) but lack reconnection support (P3.4 GAP).
- Dead event types (defined but never emitted) are a coverage gap under P3.3, not just technical debt. They create a false contract — consumers expect events that never arrive.
- For systems that use logging as their primary observability mechanism, structured log entries at lifecycle transitions may partially satisfy P3.3 — but only if they follow a consistent schema and could feasibly be consumed by external tools. Ad-hoc log lines do not qualify.
